## 1. Architecture Compliance

### CLI unchanged?

**Verified.** The spec (Section 2, Section 9) explicitly states: "No modifications to CLI commands beyond `serve` flags." Section 4.4 adds only `--portfolio` and `--read-only` as new `serve` subparser arguments. Section 9 "Files NOT to Touch" lists `ontos/commands/map.py`, `doctor.py`, `hook.py`, `init.py`. The existing CLI test suite is required to pass unchanged (Section 7). The actual codebase `ontos/cli.py` entry point is only modified for the `serve` subparser.

**Evidence:** Spec Section 9 exclusion list, Section 4.4 lines 366-373, Section 7 backward compat table.

### v4.0 tools unchanged?

**Verified.** Section 4.10 states: "Adding `workspace_id` to existing tool signatures must NOT change their output format." Section 7 guarantees: "All 8 v4.0 MCP tools work identically." Section 9: "No modifications to existing 8 v4.0 tool implementations -- Only add optional `workspace_id` parameter." The existing `test_server_integration.py` test `test_server_lists_all_eight_tools_with_correct_annotations` confirms 8 tools are expected. The spec requires this test to pass unchanged.

**Evidence:** Spec Section 4.10 constraints, Section 7 table, Section 9 exclusion list.

### stdio preserved?

**Verified.** Section 1: "all over stdio transport." Section 2 out-of-scope: "HTTP/Streamable HTTP transport." Section 4.4 continues using `server.run(transport="stdio")`. The current codebase `server.py:77` calls `server.run(transport="stdio")` and the spec preserves this.

**Evidence:** Spec Section 1, Section 2, Section 4.4.

### MCP optional?

**Verified.** Section 3 dependencies table: Python (base) remains `>=3.9`, Python (MCP extra) becomes `>=3.10`. Section 7: "`pip install ontos` (no extras) works on Python 3.9+" and "No new base dependencies added." Section 8 dependencies: "No new PyPI dependencies beyond v4.0." The actual `pyproject.toml` confirms `mcp` and `pydantic` are in `[project.optional-dependencies]` under `mcp = [...]`, not in `dependencies`.

**Evidence:** Spec Section 3, Section 7, Section 8. Codebase `pyproject.toml:40-43`.

### Layer boundaries respected?

**Verified.** The spec creates new files only in `ontos/mcp/` (portfolio.py, portfolio_config.py, scanner.py, bundler.py, write_tools.py, locking.py). It modifies `ontos/mcp/server.py`, `ontos/mcp/tools.py`, `ontos/mcp/schemas.py`, and `ontos/cli.py`. It does NOT add imports of `ontos.mcp` into `ontos/core/` or `ontos/io/`. The bundler uses existing core functions (`estimate_tokens` from `ontos/core/tokens.py`, `create_snapshot` from `ontos/io/snapshot.py`) -- this is the correct direction (mcp -> core/io, never core -> mcp). The `test_parity.py:test_import_boundaries` test enforces this and is required to pass.

**Evidence:** Spec Section 4 file tables, Section 9 "Files NOT to Touch." Codebase `tests/mcp/test_parity.py:36-42`.

## 9. Issues Found

| # | Severity | Section | Issue Description | Deviation Type |
|---|----------|---------|-------------------|---------------|
| 1 | **Critical** | 4.1 vs 6.1 | **FTS5 schema defined twice with contradictory column lists.** Section 4.1: `fts_content(title, concepts, body)`. Section 6.1: `fts_content(doc_id, workspace, title, body, concepts)`. An implementer cannot determine which is correct. The Section 6.1 version adds non-searchable metadata columns (doc_id, workspace) that inflate the FTS index without value. The BM25 weight vector `bm25(10.0, 3.0, 1.0)` only covers 3 columns but Section 6.1 has 5. This will cause incorrect ranking or a runtime error. | Internal contradiction |

## 10. Verdict

**Request Changes**

The spec is thorough, well-structured, and demonstrates strong alignment with the proposal and research on most fronts. The architecture constraints are preserved, backward compatibility is carefully maintained, and the research consensus items are implemented concretely with correct technical detail. The three diagrams are consistent with the codebase architecture.

However, three critical issues block approval:

1. **The SQLite schema is defined twice with contradictory column lists** (FTS5 table columns, `documents.body` column presence, BM25 weight vector). These are fixable issues. Once the schema is unified to a single authoritative definition (recommend Section 4.1 as the implementation section, with Section 6 referencing it rather than re-defining), and the `verify` subcommand is either added to scope or explicitly deferred with rationale, the spec should pass alignment review.
