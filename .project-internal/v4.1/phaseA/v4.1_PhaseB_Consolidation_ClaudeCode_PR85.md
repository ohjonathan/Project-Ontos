## 2. Blocking Issues

Issues that must be resolved before implementation can begin. Ordered by cross-reviewer convergence (strongest signal first).

### B-1: FTS5 Schema Is Broken — `concepts` Column Does Not Exist
**Flagged by:** R1 (#1), R2 (#1, #2, #3), R3 (#1, #3, Additional Attack A)
**Sections:** 4.1, 6.1
**Category:** Internal contradiction / runtime error

The FTS5 virtual table references a `concepts` column that does not exist in the `documents` table. Additionally, the spec contains two contradictory FTS5 schema definitions:
- Section 4.1: `fts_content(title, concepts, body)` — 3 columns
- Proposal Section 6.1: `fts_content(doc_id, workspace, title, body, concepts)` — 5 columns

The BM25 weight vector `bm25(10.0, 3.0, 1.0)` provides 3 weights, which is correct for the Section 4.1 version but wrong for the 5-column version. The `documents` table also has conflicting definitions (Section 4.1 includes `body`; Section 6.1 omits it).

R3 identified a deeper issue: FTS5 external content mode (`content='documents'`) maps columns by ordinal position, not by name. The `documents` table's ordinals (id=0, workspace=1, type=2...) do not align with FTS5's declared columns (title=0, concepts=1, body=2), causing `snippet()` to return data from the wrong columns at runtime.

**Action required:** Consolidate to a single canonical schema in Section 4.1. Add a `concepts TEXT` column to the `documents` table (populated from frontmatter during `rebuild_workspace()`). Remove the duplicate schema from Section 6.1 (reference Section 4.1 instead). Either use standalone FTS5 (not external content mode) to avoid ordinal mapping issues, or align ordinals explicitly. Fix BM25 weights to match final column count. **Estimated effort:** 1-2 hours.

### B-2: `rename_document()` Spec Claims Nonexistent Functionality
**Flagged by:** R1 (#3), R3 (#2, #15, Attack 6)
**Sections:** 4.8.2, 5 (OQ-2)
**Category:** Factual error / scope ambiguity

Two related issues:

(a) The spec references `_compute_rename_plan()` as an entry point in `ontos/commands/rename.py`. This function does not exist. The actual function is `_prepare_plan()` (line 278) with a different signature and return type (`Tuple[Optional[_PreparedPlan], Optional[RenameError]]`). R1 flagged that a developer cannot implement this section without significant reverse-engineering.

(b) R3 read all 1,362 lines of `rename.py` and found: **the existing CLI rename command does NOT perform filesystem file renames.** It only renames document IDs in frontmatter and updates references. There are zero calls to `os.rename()`, `Path.rename()`, or `SessionContext.buffer_move()` in the entire file. The spec's OQ-2 recommendation (B: "full rename when file name matches") claims "the existing `ontos rename` CLI command already handles file renaming" — this is factually incorrect.

The `RenameDocumentResponse` schema includes `old_path` and `new_path` fields for a feature that does not exist in the codebase.

**Action required:** (1) Correct the function reference from `_compute_rename_plan()` to `_prepare_plan()` and specify the adaptation strategy. (2) Resolve OQ-2 — see the OQ-2 position below. **Estimated effort:** 1 hour for the function fix; OQ-2 resolution is a design decision.

### B-3: `SessionContext.commit()` Atomicity Overstated + Dual Lock Gap
**Flagged by:** R3 (#4, #5, Attack 1, Attack 2)
**Sections:** 4.8, 4.9
**Category:** Safety / correctness

R3 verified `SessionContext.commit()` (context.py lines 158-201) and found it uses sequential temp-then-rename in Phase 2. If the process crashes after renaming 3 of 7 files, already-renamed files are NOT rolled back. The spec calls this "atomic multi-file writes" but it is actually "best-effort sequential commit with temp-file cleanup."

Additionally, R3 discovered that the MCP workspace lock (`.ontos.lock`) and CLI write lock (`.ontos/write.lock`) are completely different files with zero coordination. Simultaneous `ontos rename` (CLI) and `rename_document()` (MCP) would proceed concurrently and corrupt workspace state. The spec acknowledges this risk in Section 8 but provides no mitigation.

R3 also noted the CLI rename command requires a clean git state (`_check_clean_git_state()`) but the MCP tool spec has no such precondition.

**Action required:** (1) Downgrade the atomicity claim to "best-effort sequential commit." (2) Add git clean-state precondition for `rename_document()`. (3) Document the recovery path (`git checkout -- .`). (4) Either unify the MCP and CLI lock files, or have MCP write tools acquire both locks. **Estimated effort:** 2 hours.

### B-4: `get_context_bundle()` Crashes in Single-Workspace Mode
**Flagged by:** R3 (#6, Attack 5)
**Section:** 4.7
**Category:** Logic error

When `workspace_id` is passed in single-workspace mode (where `portfolio_index is None`), the code falls to the `else` branch (line 784) and calls `_validate_workspace_id(portfolio_index, workspace_id)`. Since `portfolio_index is None`, this causes `AttributeError`. This contradicts Section 4.10's contract that `workspace_id` is "accepted but ignored" in single-workspace mode.

R1 independently noted the UX concern about `workspace_id`/`workspace_slug` confusion in the response schema but did not trace the crash path.

**Action required:** Add a guard at the top of `get_context_bundle()` and all portfolio tools: if `portfolio_index is None` and `workspace_id is not None`, either ignore `workspace_id` (per Section 4.10 contract) or return `isError: true` with `E_CROSS_WORKSPACE_NOT_SUPPORTED`. **Estimated effort:** 30 minutes.

### B-5: `_invoke_tool()` Routing Logic Underspecified
**Flagged by:** R1 (#2)
**Section:** 4.4
**Category:** Design gap

The current `_invoke_tool()` passes a `SnapshotCacheView` as the first positional arg to `tool_fn`. For portfolio tools, it needs to pass `portfolio_index` instead. The spec adds `portfolio_index` as a keyword parameter but never specifies how the function decides which object to pass. R1 recommends creating separate `_invoke_portfolio_tool()` and `_invoke_write_tool()` wrappers.

**Action required:** Specify the routing mechanism — either separate invoke functions per tool category, a flag-based dispatch, or passing both objects via kwargs. R1's recommendation for separate wrappers is the cleanest. **Estimated effort:** 30 minutes.

### B-6: `verify` Subcommand Silently Omitted from Proposal D8
**Flagged by:** R2 (#4)
**Section:** N/A (missing from spec)
**Category:** Silent proposal deviation

Proposal D8 specifies a `verify` subcommand that "compares the DB against `projects.json` and reports discrepancies" as part of the gradual `.dev-hub` transition. The spec does not mention this subcommand anywhere and does not acknowledge the omission. The 2-week transition criterion from the proposal is also silently dropped. R2 is the only reviewer who caught this — R1 and R3 did not check proposal D8 alignment.

**Action required:** Either add `verify` to the Track A scope, or explicitly defer it with rationale in the Exclusion List (Section 9). Silent omission is not acceptable for an approved proposal decision. **Estimated effort:** 15 minutes for deferral note; 2-4 hours if implementing.

## Open Question Positions

### OQ-1: `get_context_bundle()` in single-workspace mode

**Position: (B) Available in both modes — with the logic fix from B-4.**

Rationale: `get_context_bundle()` provides token-budgeted JSON output that `context_map` does not. The differentiation is genuine. The logic error (B-4) is a localized code fix, not a design flaw. R3's attack on option (B) found the logic error but concluded the recommendation is sound after the fix. The tool adds value in single-workspace mode (token budgeting, structured output, staleness detection).

### OQ-2: Rename tool — file path rename vs. ID-only rename

**Position: (A) ID-only rename.**

Rationale: R3's codebase evidence is decisive — the existing CLI rename command performs ID-only rename with zero filesystem rename operations. The spec's recommendation of (B) was based on a factual error ("The existing `ontos rename` CLI command already handles file renaming"). Implementing file rename would be entirely new functionality requiring: path construction rules, case-sensitivity handling (macOS APFS), date-prefix conventions for logs, and git history considerations. This is an unacknowledged scope increase that pushes Track B risk from MEDIUM-HIGH to HIGH.

Recommend (A) for v4.1. If file rename is desired, scope it as a v4.2 feature with dedicated design. Remove `old_path`/`new_path` from `RenameDocumentResponse` (or keep them identical since path doesn't change).
