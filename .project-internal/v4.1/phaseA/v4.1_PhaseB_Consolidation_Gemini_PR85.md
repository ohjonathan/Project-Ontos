## 2. Blocking Issues

- **ID:** B-01
  - **Description:** Missing `verify` subcommand in the spec and `refresh()` tool update details omitted. FTS5 external content sync strategy needs explicit sequencing (deleting FTS rows before content rows).
  - **Flagged by:** R1 (Peer)
  - **Category:** Completeness / Implementability
  - **Action Required:** Update spec to include the `verify` subcommand, detail `refresh()`, and explicitly sequence the FTS5 external content sync.

- **ID:** B-02
  - **Description:** Write tool atomicity risk. `SessionContext.commit()` is not truly atomic across multiple files (e.g., in `rename_document`). A crash mid-commit results in a partially updated workspace with no rollback.
  - **Flagged by:** R3 (Adversarial)
  - **Category:** Data Integrity / Atomicity
  - **Action Required:** Define a rollback mechanism or true cross-file atomic write guarantees for multi-file operations.

- **ID:** B-03
  - **Description:** Workspace file locking vulnerability. The proposed PID-based locking system is susceptible to PID-reuse issues and does not protect against concurrent CLI operations. Minor duplication in locking logic between new `workspace_lock` and existing `SessionContext` lock.
  - **Flagged by:** R1 (Peer), R3 (Adversarial)
  - **Category:** Concurrency / Locking
  - **Action Required:** Redesign the locking mechanism to be robust against PID reuse and concurrent CLI operations, and consolidate duplicate locking logic.

- **ID:** B-04
  - **Description:** `get_context_bundle()` ranking determinism. The algorithm lacks a deterministic tie-breaking rule, which can lead to prompt instability and degraded LLM cache performance.
  - **Flagged by:** R3 (Adversarial)
  - **Category:** Determinism / Caching
  - **Action Required:** Add explicit deterministic tie-breaking rules (e.g., sort by path alphabetically after rank) to the context bundle algorithm.

## 3. Should-Fix Issues

- **ID:** S-01
  - **Description:** Track A -> B handoff assumptions. Ensure Track B explicitly states what state it assumes Track A will leave the system in, so Track B can be reviewed in isolation.
  - **Flagged by:** R3 (Adversarial)
  - **Category:** Developer Experience
  - **Action Required:** Clarify the handoff state between Track A and Track B in the spec.

## 4. Minor Issues

- None identified that do not fall into Blocking or Should-Fix. The spec generally aligns well with the research consensus and proposal.

## 5. Agreement Analysis

- **Strong Agreement:** All reviewers agree that the spec aligns well with the research consensus (e.g., SQLite PRAGMA, FTS5 configuration, `BEGIN IMMEDIATE`) and adheres to the stdio-only re-scoped roadmap.
- **Single-Reviewer Findings:** R3's identification of the `SessionContext.commit()` non-atomicity and lack of deterministic tie-breaking in `get_context_bundle()` are critical catches that were not highlighted by R1 or R2. R1 identified the missing `verify` subcommand.
- **Direct Disagreements:** R2 (Alignment) approved the spec based on strict alignment with the proposal, while R1 and R3 requested changes due to implementation-level gaps (atomicity, locking, missing subcommands). This is an expected difference in lens, highlighting that while the architecture aligns, the low-level implementation details need hardening.

## 6. Required Actions for CA

1. **Address Write Tool Atomicity (Effort: High):** Redesign or clarify the `SessionContext.commit()` mechanism to guarantee multi-file atomicity or specify a clear rollback procedure.
2. **Robust Workspace Locking (Effort: Medium):** Replace the PID-based locking system with a more robust mechanism that prevents CLI/MCP concurrency issues and PID reuse problems. Consolidate with existing `SessionContext` lock.
3. **FTS5 Sync Sequencing (Effort: Low):** Explicitly define the sequence for FTS5 synchronization (delete FTS rows before content rows).
4. **Deterministic Tie-Breaking (Effort: Low):** Add deterministic sorting (e.g., secondary sort by path) to `get_context_bundle()`.
5. **Add Missing Components (Effort: Low):** Document the `verify` subcommand and `refresh()` tool details.

## 7. Decision Summary

**Verdict:** Needs Revision

### Position on OQ-1 and OQ-2
- **OQ-1 (Tool Error Response Strategy):** Recommend proceeding with the `isError: true` pattern as aligned with the research consensus. Throwing exceptions breaks the LLM's ability to interpret and recover from the error gracefully in standard MCP clients.
- **OQ-2 (FTS5 Rebuild Concurrency):** Recommend enforcing strict sequential locking during rebuilds rather than relying purely on `BEGIN IMMEDIATE` if Track B writes can interleave unpredictably. The risk of partial reads by LLMs during a long rebuild outweighs the brief pause in read availability.
