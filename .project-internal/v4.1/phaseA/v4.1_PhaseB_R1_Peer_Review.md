## 2. Developer Test (Implementability)

For each Technical Design section, answering: "Could a developer implement this without asking the spec author a question?"

### 4.1 Portfolio DB Layer -- MOSTLY YES

A developer could build this. The schema, PRAGMAs, concurrency rules, and class interface are all specified. Two questions would arise:

1. **FTS5 schema mismatch:** The schema in Section 4.1 defines the FTS5 table with columns `(title, concepts, body, content='documents', content_rowid=rowid, ...)` but the `documents` table does NOT have a `concepts` column. Where does `concepts` come from when populating FTS rows? The developer would need to ask: "Do I extract concepts from frontmatter during `rebuild_workspace()`?"
2. **`_connect()` method:** The `search_fts()` code sample calls `self._connect()` but this method is not defined in the class interface. The developer would infer it creates a new SQLite connection (per the "no pooling" rule), but it should be explicit.

### 4.2 Portfolio Config -- YES

Straightforward dataclass + TOML loading. The `tomli`/`tomllib` conditional import is already established in the codebase.

### 4.3 Portfolio Scanner -- MOSTLY YES

The discovery algorithm is clear but the `partial` classification heuristic has ambiguity: "readme or ontos with <5 docs." Does this mean `(has_readme AND NOT has_ontos) OR (has_ontos AND doc_count < 5)`? Or `(has_readme OR has_ontos) AND doc_count < 5`? A developer would need to ask.

### 4.4 Server Bootstrap -- YES (with caveats)

The registration refactoring plan is clear. However, the `_invoke_tool()` modification (adding `portfolio_index` parameter) creates a design tension: the function currently passes `tool_input` (a `SnapshotCacheView`) as the first arg to `tool_fn`. For portfolio tools, it needs to pass `portfolio_index` instead. The spec says "For portfolio tools, the `tool_fn` receives `portfolio_index` as its first argument instead of `cache`" but does not specify how `_invoke_tool()` decides which to pass. A conditional? A flag? This matters for implementation.

### 4.5 project_registry -- YES

Simple, clean, no ambiguity.

### 4.6 search -- YES

Comprehensive: query sanitization logic, workspace validation, FTS5 SQL with BM25, pagination. The `_sanitize_fts_query()` function is described at the algorithm level (strip, escape, detect operators) but a developer could implement it.

### 4.7 get_context_bundle -- MOSTLY YES

The 6-step algorithm is detailed. Two small gaps:

1. **In-degree calculation for non-portfolio mode:** The spec says `in_degree = len(snapshot.graph.reverse_edges.get(doc.id, []))`. In single-workspace mode, the tool receives a `SnapshotCacheView` (which has `snapshot`), so this works. But the code sample in Section 4.7 step 3 shows `get_context_bundle(portfolio_index, cache, ...)` -- in single-workspace mode, `portfolio_index` would be `None`. The spec addresses this but the code path for building a snapshot from a non-primary workspace in portfolio mode is complex and only partially specified.
2. **`build_context_bundle()` signature takes `DocumentSnapshot`** but the staleness detection in Step 6 references `ontos/core/staleness.py`. What function from that module? What's its signature?

### 4.8 Write Tools -- MOSTLY YES

- **4.8.1 scaffold_document:** Clear enough. The mapping from `create_scaffold()` (which takes a `Path`) to the MCP tool (which takes explicit type/id/title) is well-explained. The step about path determination ("subdirectory based on type if the project uses type-based organization") is slightly vague.
- **4.8.2 rename_document:** The spec references `_compute_rename_plan()` but this function does not exist in the codebase. The actual entry point is `_prepare_plan()` (line 278 of rename.py) which returns a `_PreparedPlan`. The spec says "Adapt the rename command's internal logic" but the rename command's logic is deeply integrated with the CLI -- it uses `RenameOptions`, `_LoadedScope`, and interactive confirmation patterns. A developer would need to reverse-engineer the internals, not just read this spec section. This is a critical gap.
- **4.8.3 log_session:** Clear. The `create_session_log()` call matches the actual function signature (line 64-68 of log.py).
- **4.8.4 promote_document:** Clear. The `get_curation_info()` function is at line 304 (not 22-28 as the spec claims -- see Issue #5), but the signature and usage pattern are correct.

### 4.9 Workspace Locking -- YES

Full implementation provided, edge cases addressed.

### 4.10 workspace_id Threading -- YES

Simple parameter additions, backward-compat plan clear.

### 4.11 Pydantic Schemas -- YES

All models listed, consistent with `StrictModel` pattern.

## 5. Test Strategy Assessment

### Is ~139 new tests sufficient?

For 7 new tools + 1 new DB layer + 1 new config layer + 1 new scanner + 1 locking module, 139 tests is reasonable but not generous. By comparison, the existing 8 v4.0 tools have ~100 MCP tests across 13 files. Adding 7 tools + significant infrastructure should proportionally require more.

### Coverage Gaps

1. **No FTS5 edge case tests:** The spec lists `test_portfolio.py` with ~15 tests but does not mention testing FTS5 with non-ASCII content (porter unicode61 tokenizer), very long documents, or documents with no body content. These are important for a search feature.

2. **No portfolio scanner error tests:** What happens when `registry_path` points to a corrupt JSON file? When a `scan_root` doesn't exist? When a `.git/` directory is a file (submodule) not a directory? `test_scanner.py` lists ~8 tests but the descriptions don't cover error paths.

3. **No test for `_sanitize_fts_query()` edge cases:** FTS5 query injection is listed as a MEDIUM risk in Section 8, but the test plan doesn't explicitly mention testing malformed FTS5 queries (unmatched quotes, injection attempts like `"") VALUES("evil`).

4. **No concurrent write test:** `test_locking.py` lists "concurrent access simulation" (~6 tests) but the spec doesn't describe how concurrency is simulated in the test. Using `threading.Thread`? `multiprocessing.Process`? This matters because the lock is PID-based.

5. **No test for `promote_document` returning `success: false`:** The test plan mentions "validation failure (missing fields)" but doesn't explicitly verify the `success: false` path where the response is NOT `isError: true`. This is the most unusual error handling pattern in the spec.

6. **Missing: Cross-workspace `workspace_id` rejection test.** Section 4.10 says non-primary `workspace_id` in portfolio mode returns `E_CROSS_WORKSPACE_NOT_SUPPORTED`. No test is listed for this path.

### Integration Test Adequacy

The integration tests cover the main cross-tool workflows (scaffold -> list, scaffold -> rename -> list, etc.) and the portfolio mode tool chain (registry -> search -> bundle). This is adequate for the happy path.

**Gap:** No integration test for write tool + portfolio index invalidation under concurrent reads. If one tool is writing while another is reading from the portfolio DB, the `BEGIN IMMEDIATE` + WAL mode should handle this, but it's not tested.

### Manual Testing Plan

The manual testing steps (Section 6) cover 14 scenarios across Track A and Track B. This is adequate. The most critical manual test is step 3 in Track B (rename with reference updates) -- this exercises the riskiest code path.

## 6. UX Review

### Tool Names

The tool names are intuitive:
- `project_registry` -- clear what it returns
- `search` -- generic but works in context
- `get_context_bundle` -- slightly verbose but descriptive
- `scaffold_document`, `rename_document`, `log_session`, `promote_document` -- verb_noun pattern, clear

**Concern:** `search` is very generic. In a multi-server MCP environment, an LLM might confuse `search` with other search tools from other servers. However, the MCP server name ("Ontos") provides disambiguation. Minor concern.

### Parameter Consistency

| Parameter | Used In | Consistent? |
|-----------|---------|-------------|
| `workspace_id` | All tools (optional) | Yes -- consistently `Optional[str] = None` |
| `document_id` | rename, promote, query | Yes |
| `query_string` | search | OK -- disambiguates from FTS5 "query" concept |
| `token_budget` | get_context_bundle | Yes |
| `offset` / `limit` | search, list_documents | Yes -- same semantics |

**Inconsistency found:** In `get_context_bundle()`, the code sample (Section 4.7 step 3) shows `workspace_id: Optional[str] = None` as the parameter name. But in the response schema `GetContextBundleResponse`, the field is called `workspace_id` AND `workspace_slug` (both present). The distinction between these two is unclear: are they always identical? The slugification rule says `slug = slugify(directory_name)`, so `workspace_id` IS the slug. Having both fields is confusing.

### Error Messages

Error messages follow a good pattern: what happened + why + how to fix. Examples:
- `E_UNKNOWN_WORKSPACE`: Lists valid slugs, points to `project_registry()`
- `E_WORKSPACE_BUSY`: Explains what's happening, suggests retry
- `E_UNDOCUMENTED_WORKSPACE`: Directs to `ontos init`

This is excellent. Follows the research's recommendation for actionable tool errors (Area 2).

### `--read-only` Flag Behavior

Well-specified: write tools are entirely stripped from `tools/list`. Not just disabled -- absent. This matches the GitHub MCP server pattern cited in the research. Clean.

### `promote_document` returning `success: false`

This is the most subtle UX decision in the spec. When `promote_document` fails validation (e.g., missing L2 fields), it returns:
```json
{"success": false, "document_id": "...", "validation_warnings": ["..."]}
```
NOT `isError: true`.

**Problem:** This means the LLM receives a successful MCP response (no `isError` flag) containing a semantic failure. The LLM must inspect the `success` field to understand the outcome. This is inconsistent with the other write tools, which all use `isError: true` for failures. An LLM optimizing for `isError` detection might miss this.

**However:** The spec's rationale is sound -- this is a "normal" response (the tool correctly determined the document can't be promoted), not an error. The `validation_warnings` list tells the LLM exactly what's missing. This matches the proposal's design (Section 5.2).

**Recommendation:** Keep the design, but add an explicit note in the tool description that `success: false` is a valid response indicating promotion requirements are not met. The LLM should check `success` before celebrating.

## 7. Quality Assessment

### Track A: HIGH QUALITY

The portfolio DB design is well-researched and leverages all 10 research consensus items correctly. The PRAGMA block, FTS5 configuration, concurrency rules, and rebuild strategy are all copy-paste ready from the research. The tool implementations are simple and focused.

### Track B: MEDIUM-HIGH QUALITY

The write tools are well-structured with consistent error handling patterns. The locking mechanism is simple and correct. The main quality concern is the `rename_document()` implementation section, which references non-existent internal functions and doesn't provide a clear adaptation path from the CLI rename logic to the MCP tool.

### Shared: HIGH QUALITY

The registration plumbing changes are mechanical and well-specified. The `workspace_id` threading plan is careful about backward compatibility.

### Pattern Consistency

Patterns are highly consistent:
- All tools follow `def tool_name(cache_or_index: Any, *, params) -> dict[str, Any]`
- All errors use `OntosUserError` with error codes
- All responses validated through Pydantic schemas
- All write tools use the workspace lock -> execute -> refresh pattern
- Registration uses the same `register(name=, title=, description=, handler=, annotations=, meta=)` closure

### Overcomplicated Areas

1. **`_invoke_tool()` modification (Section 4.4):** Adding `portfolio_index` as a parameter to `_invoke_tool()` creates a function that does too many things. It's already handling freshness, live cache mode, logging, error catching, and schema validation. Adding portfolio routing makes it a god function. Consider separate `_invoke_portfolio_tool()` and `_invoke_write_tool()` wrappers instead. This would also solve the "how does it know which first arg to pass" ambiguity.

2. **FTS5 content sync (Section 4.1 / OQ-3):** The manual sync approach (delete all, reinsert all per workspace) is correct but the FTS5 content table needs careful handling of the `DELETE` command (`INSERT INTO fts_content(fts_content, rowid, ...) VALUES('delete', ?, ?, ?, ?)`). The spec describes this in the Note after the schema but a developer might miss it. The actual SQL for the delete-then-reinsert pattern should be in the `rebuild_workspace()` steps, not just in a note.

## 10. Verdict

**Request Changes**

The spec is substantially well-crafted -- the research integration, backward-compatibility discipline, and Track A design are all strong. However, three critical issues block implementation as-is:

1. The FTS5 schema references a `concepts` column that doesn't exist in the `documents` table, which will cause a runtime error. This is a mechanical fix but must be done before implementation begins.
2. The `_invoke_tool()` modification for portfolio tools is architecturally underspecified -- a developer cannot implement the routing logic from what's written.
3. The `rename_document()` section references a non-existent internal function and doesn't provide a viable adaptation path from the CLI rename module to the MCP tool. Given that rename is the highest-risk write tool (the spec itself acknowledges this), the implementation path must be concrete.

These three issues are fixable in a focused revision pass without restructuring the spec. The remaining 21 issues are Major/Minor/Nit and would benefit from being addressed but do not block implementation.

**Confidence: High.** I read every line of the spec, cross-referenced against 14 source files, and verified line numbers, function signatures, and patterns against the actual codebase. The issues identified are based on concrete mismatches, not hypothetical concerns.
