## 1. Attack Results

### Attack 1: Write Tool Atomicity (Section 4.8)

**Attack description:** The spec claims write tools use `SessionContext` for atomic multi-file writes and that `rename_document()` wraps `ontos/commands/rename.py`. I read every line of both `SessionContext.commit()` and `rename_command()` to verify whether "atomic" means what the spec implies.

**Finding:** `SessionContext.commit()` does NOT provide all-or-nothing atomicity. Here is what actually happens:

1. **Phase 1** (lines 159-172 of `ontos/core/context.py`): Write all content to `.tmp` sidecar files.
2. **Phase 2** (lines 175-184): Iterate through staged pairs and rename each temp file to its final path.

If the process crashes after renaming 3 of 7 files in Phase 2, the first 3 files are written with their new content, the remaining 4 are untouched, and the temp files for unprocessed items are cleaned up by the exception handler (lines 186-194). But the already-renamed files are NOT rolled back. The exception handler only cleans up temp files that still exist -- it does not restore original content of files that were already overwritten by `temp.rename(final)`.

This means `rename_document()` modifying 7 documents (1 target + 6 referencing docs) can leave the workspace in a state where some documents reference the new ID and others still reference the old ID. The spec calls this "atomic multi-file writes" (Section 4.8.2), but it is actually "sequential writes with temp-file cleanup on failure."

Additionally, the existing `rename_command()` in `ontos/commands/rename.py` (line 340) **requires a clean git state** before applying via `_check_clean_git_state()`. The MCP tool has no such precondition mentioned in the spec. Without a clean git state, the user's safety net (`git checkout` to recover) is degraded because they cannot distinguish rename-caused changes from pre-existing uncommitted changes.

The rename command's own error message at line 235 acknowledges this: `"commit failed after staging; repository may be partially updated"`.

**Severity:** Major

**Evidence:**
- `/Users/jonathanoh/Dev/Ontos-dev/ontos/core/context.py` lines 158-201: Phase 2 is not transactional. Each `temp.rename(final)` succeeds individually. No journal, no WAL, no undo log.
- `/Users/jonathanoh/Dev/Ontos-dev/ontos/commands/rename.py` lines 219-224: `ctx = SessionContext.from_repo(...)`, `ctx.buffer_write(...)`, `ctx.commit()` -- uses only `buffer_write`, never `buffer_move` for file renames.
- `/Users/jonathanoh/Dev/Ontos-dev/ontos/commands/rename.py` lines 339-346: CLI requires clean git state; MCP spec does not.

**Recommendation:**
1. Downgrade the atomicity claim in Section 4.8.2 from "apply all edits atomically using SessionContext" to "apply all edits using SessionContext's best-effort commit (sequential temp-then-rename, with cleanup on failure)."
2. Add a precondition check in `rename_document()` that verifies git working tree is clean before executing. If dirty, return `isError: true` with guidance to commit or stash first.
3. Add the partial-commit failure mode to the risk table: "rename may leave workspace with mixed old/new references if interrupted mid-commit."
4. Document that `git checkout -- .` is the recovery path.

### Attack 2: Workspace File Locking (Section 4.9)

**Attack description:** Systematic attack on the `workspace_lock()` implementation for race conditions, PID reuse, zombies, and CLI interaction gaps.

**Finding: PID Reuse Attack (Low probability, real impact)**

On macOS, PIDs wrap at 99998. After a stale lock is detected (PID is dead), the code calls `lock_path.unlink(missing_ok=True)` at line 1139 and `continue`s to retry `os.open()`. Between the `unlink()` and the retry, another MCP server process could have acquired the lock. But the more insidious case: the stale PID could have been reused by an unrelated process (e.g., a Chrome renderer). `os.kill(pid, 0)` succeeds, so the lock appears held by a live process. The lock becomes unbreakable until that unrelated process exits.

Probability: Low but non-zero. With 99998 PIDs and typical macOS PID recycling, a 10-second-old stale lock from a PID that got reused requires ~100K process forks in 10 seconds. Unlikely. But a lock from a previous day (user killed the server, forgot to clean up) combined with a reused PID is plausible.

**Finding: TOCTOU Window in Stale Lock Breaking**

Between reading the PID (line 1133), checking liveness (line 1134), and unlinking (line 1139), another process could:
1. Read the same stale lock
2. Also decide to break it
3. Both unlink and both re-enter the `os.open` loop
4. Both succeed in creating the lock (but only one's PID gets written)

This is a TOCTOU race. The `O_CREAT | O_EXCL` on re-entry prevents both from creating the lock file, but the timing window between unlink and re-open means one process might see `FileExistsError` from the other's freshly created lock and misidentify it as stale (because it was just created and the PID matches a live process). This is self-correcting -- the second process waits and retries correctly. But the first retry after unlink is a wasted cycle.

**Finding: CLI Concurrent Writes (Acknowledged but Unmitigated)**

The spec explicitly acknowledges this in the Section 8 risk table: "CLI commands don't acquire `.ontos.lock`." The `SessionContext._acquire_lock()` in `ontos/core/context.py` uses a *different* lock file: `.ontos/write.lock` (line 148), not `.ontos.lock`. So the MCP workspace lock and the CLI write lock are completely independent. Simultaneous `ontos rename` (CLI, acquires `.ontos/write.lock`) and `rename_document()` (MCP, acquires `.ontos.lock`) will proceed concurrently with no coordination and corrupt workspace state.

**Finding: Lock File in Git**

`.ontos.lock` is not in `.gitignore`. The spec says to "Add `.ontos.lock` to the `.gitignore` pattern if not already present" (line 1171), but this is a TODO, not an implementation step. If a developer commits `.ontos.lock` accidentally (e.g., via `git add -A`), subsequent `git checkout` of a branch containing it would recreate the file. The PID in the file would be from the developer's session at commit time, not a live process. Stale detection would break it, but this is an avoidable confusion.

**Finding: SIGKILL Lock Persistence**

If a process receives SIGKILL, the `finally` block (line 1151-1154) never executes. The lock persists until another process detects it as stale. This is documented behavior for PID-based locks and is correctly handled by the stale detection logic. The spec's design handles this correctly.

**Severity:** Minor (PID reuse), Minor (TOCTOU), Major (CLI concurrent writes), Minor (gitignore)

**Evidence:**
- `/Users/jonathanoh/Dev/Ontos-dev/ontos/core/context.py` line 148: CLI uses `.ontos/write.lock`
- Spec Section 4.9 lines 1119-1154: MCP uses `.ontos.lock`
- These are different locks with zero coordination.

**Recommendation:**
1. **Critical:** Make the MCP workspace lock and CLI write lock the same file, OR have the MCP lock also acquire `.ontos/write.lock` as a nested lock. This is the only way to prevent concurrent CLI + MCP writes.
2. Add `.ontos.lock` to `.gitignore` as an explicit implementation step in the file table, not a "Constraints" note.
3. Consider writing PID + process start time to the lock file (not just PID) to mitigate PID reuse. `os.getpid()` + `psutil.Process(os.getpid()).create_time()` (but psutil is a new dep). Alternatively, store a random nonce alongside PID, or use `fcntl.flock()` which is immune to PID reuse.

### Attack 3: FTS5 Rebuild Atomicity Under Concurrent Reads

**Attack description:** `rebuild_workspace()` uses `BEGIN IMMEDIATE`, deletes all FTS rows for a workspace, then reinserts. What happens to concurrent `search()` queries?

**Finding: WAL mode protects readers**

With WAL mode, `BEGIN IMMEDIATE` blocks other writers but NOT readers. Readers see the pre-transaction snapshot until the writer commits. So a `search()` query arriving during `rebuild_workspace()` will see the old data, not empty results. This is correct behavior.

However, there is a subtlety with FTS5 external content mode. The FTS5 virtual table with `content='documents'` stores its own inverted index but reads column data from the `documents` table at query time. During `rebuild_workspace()`:

1. `DELETE FROM documents WHERE workspace = ?` executes
2. `INSERT INTO fts_content(fts_content, rowid, ...) VALUES('delete', ...)` removes FTS entries
3. New rows are inserted into `documents` and `fts_content`

A reader inside a WAL snapshot will see the old `documents` rows (correct) and the old FTS index (correct). The WAL isolation holds. But after commit, there is a brief window where the FTS index is freshly rebuilt and the `documents` table has new rows. The `snippet()` function reads from `documents` at query time. If a query starts just before commit and the FTS match is against a row whose `rowid` changed (because delete+insert can assign different rowids), the `JOIN documents d ON d.rowid = fts_content.rowid` could join to the wrong document or miss it entirely.

Actually, on closer analysis, because WAL provides snapshot isolation at the transaction level, a reader that started its transaction before the commit will see old data consistently. A reader that starts after commit sees new data consistently. The real concern is readers that don't use explicit transactions -- each statement is its own implicit transaction. The `search_fts()` implementation (Section 4.6) executes two queries: the main search query and a COUNT query. Between these two queries (lines 577-588), the writer could commit. The main query returns old results, the COUNT returns the new count. This is a minor consistency issue.

**Finding: Write tools call `rebuild_workspace()` inside the workspace lock, but `search()` does not acquire the workspace lock.**

This is by design (search is read-only). But it means search results can be briefly stale after a write tool completes. The spec does not document this behavior.

**Severity:** Minor

**Evidence:**
- Spec Section 4.6, lines 575-588: Two separate queries without a shared transaction
- Spec Section 4.1, lines 212-218: `rebuild_workspace()` logic

**Recommendation:**
1. Wrap both the main search query and the COUNT query in a single explicit `BEGIN` (deferred is fine for reads) to ensure snapshot consistency.
2. Document in Section 4.6 constraints that search results may briefly lag behind write tool completions (acceptable behavior for a cache-based system).

### Attack 5: OQ-1 Attack (`get_context_bundle` in single-workspace mode)

**Attack description:** The spec recommends option (B): available in both modes. Attack the recommendation.

**Finding: The recommendation is sound but the implementation has a logic error.**

In the `get_context_bundle()` code (Section 4.7, lines 772-779):

```python
if workspace_id is None:
    if portfolio_index is not None:
        raise OntosUserError(
            "workspace_id is required in portfolio mode. ..."

    # Single-workspace mode: use the cache's workspace
    slug = slugify(cache.workspace_root.name)
```

This works correctly. In single-workspace mode, `portfolio_index is None`, so we fall through to using the cache. In portfolio mode, `portfolio_index is not None` and `workspace_id is None` raises an error.

But `portfolio_index` is `None` in single-workspace mode! This would cause an `AttributeError` when `_validate_workspace_id` calls `portfolio_index.get_projects()`.

The spec does not handle this case. Section 4.10 says "In single-workspace mode: workspace_id is accepted but ignored." But the `get_context_bundle()` code does NOT ignore it -- it branches on it.

**Case for (A) -- portfolio-only:** In single-workspace mode, `context_map(compact="tiered")` already returns structured orientation context. Adding `get_context_bundle()` creates a second tool that serves a similar purpose (workspace orientation). LLMs with two overlapping tools may call the wrong one. **Severity:** Major (logic error), Minor (design question)

**Evidence:**
- Spec Section 4.7, lines 772-811: `get_context_bundle()` implementation
- Spec Section 4.10, line 1188: "workspace_id is accepted but ignored" -- but the code does NOT ignore it

**Recommendation:**
1. Add an explicit guard at the top of `get_context_bundle()`: if `portfolio_index is None` and `workspace_id is not None`, either ignore `workspace_id` (per Section 4.10 contract) or return `isError: true` with `E_CROSS_WORKSPACE_NOT_SUPPORTED`.
2. Apply this same guard to all portfolio tools. Section 4.10 promises this behavior but no tool implementation enforces it.

**Severity:** Critical (spec claims nonexistent functionality)

**Evidence:**

2. **Option B: If file rename is desired,** acknowledge it as NEW functionality requiring its own design section. Address: path construction, case sensitivity, date-prefixed filenames, git impact. This adds scope to Track B.

### Attack 7: `get_context_bundle()` Ranking Determinism (Section 4.7)

**Attack description:** Probe the bundling algorithm for non-determinism and boundary effects.

**Finding: In-degree Tie-Breaking**

The scoring formula (Section 4.7, Step 1) assigns all non-kernel documents with the same in-degree the same score: `0.5 + 0.5 * (in_degree / max_in_degree)`. If `max_in_degree = 4` and two documents both have `in_degree = 2`, both get score 0.75. The sort is "in-degree desc" which places them adjacent, but their relative order depends on the iteration order of `snapshot.documents.values()`. Python dicts preserve insertion order (since 3.7), and `timsort` is stable, so the order is deterministic for a given snapshot. But if files are added or removed between runs, the insertion order can change, changing which document appears first in a tie. This affects both greedy packing (who gets included near the budget boundary) and lost-in-the-middle ordering.

This is not a bug -- it is a documented consequence of greedy algorithms. But the spec does not acknowledge it, and it means `get_context_bundle()` output can change between runs even if no document content changed (only if document set changed).

**Finding: Lost-in-the-Middle Interleaving**

The algorithm (Section 4.7, Step 4) says: "sort included by score descending. Interleave: index 0 -> position 0, index 1 -> position -1, index 2 -> position 1, index 3 -> position -2, etc."

This is underspecified. With 5 documents at scores [1.0, 0.8, 0.75, 0.75, 0.6], the interleaving produces:
- Position 0: score 1.0 (index 0)
- Position -1 (last): score 0.8 (index 1)
- Position 1: score 0.75 (index 2)
- Position -2 (second-to-last): score 0.75 (index 3)
- Position 2 (middle): score 0.6 (index 4)

This puts the two 0.75 documents at positions 1 and -2. Their relative assignment depends on sort order, which depends on insertion order for equal scores. The spec should note that equal-score documents have arbitrary (but deterministic) interleaving.

**Finding: `estimate_tokens()` Boundary Effects**

`len(content) // 4` is integer division. Content of 4095 characters = 1023 tokens. Content of 4096 characters = 1024 tokens. This 1-character difference could determine whether a document fits within the remaining budget. Combined with the "kernel documents are always included, even if they exceed 50% of budget" rule, a kernel document with 32769 characters (8192 tokens) would consume the entire default budget, leaving zero room for any other document. The spec says kernels are "always included" without an upper bound on kernel size.

For the `len(content) // 4` heuristic specifically: English text averages ~4 chars/token, but code-heavy documents (common in technical documentation) average ~3.5 chars/token. CJK text averages ~1.5-2 chars/token. A document with significant code blocks could be underestimated by 15%, causing the bundle to exceed the nominal budget. The spec acknowledges this risk in proposal Section D4 (">20% miss triggers tiktoken") but does not add a defensive margin to the budget check.

**Severity:** Minor (tie-breaking), Minor (token estimation)

**Evidence:**
- `/Users/jonathanoh/Dev/Ontos-dev/ontos/core/tokens.py` lines 8-21: `len(content) // 4`
- Spec Section 4.7, Step 3: No kernel size cap
- Spec Section 4.7, Step 4: Interleaving algorithm

**Recommendation:**
1. Add a secondary tiebreaker to the sort: document ID alphabetically. This makes the output fully deterministic regardless of insertion order.
2. Add a note that kernel documents are always included but if a single kernel exceeds the full budget, a warning is emitted in the response.
3. Consider a 10% defensive margin: actual packing budget = `token_budget * 0.9` to account for heuristic error.

## 2. Additional Attacks

### Additional Attack A: FTS5 Schema Inconsistency Between Sections

**Attack description:** Compare the two FTS5 schema definitions in the spec.

**Finding:** The spec contains TWO different FTS5 schema definitions that contradict each other.

**Section 4.1** (line 189-197):
```sql
CREATE VIRTUAL TABLE fts_content USING fts5(
    title,
    concepts,
    body,
    content='documents',
    content_rowid=rowid,
    ...
);
```
Columns: `title(0), concepts(1), body(2)` -- 3 columns.

**Proposal Section 6.1** (line 336-346):
```sql
CREATE VIRTUAL TABLE fts_content USING fts5(
    doc_id,
    workspace,
    title,
    body,
    concepts,
    content=documents,
    ...
);
```
Columns: `doc_id(0), workspace(1), title(2), body(3), concepts(4)` -- 5 columns.

The spec's Section 4.1 version has 3 columns but the BM25 rank `'bm25(10.0, 3.0, 1.0)'` provides 3 weights -- this is consistent. The proposal's version has 5 columns with 3 weights -- inconsistent (BM25 needs one weight per column).

**Critical issue with external content mode:** FTS5 with `content='documents'` maps FTS columns to content table columns BY ORDINAL POSITION, not by name. The spec's `documents` table columns are: `id(0), workspace(1), type(2), status(3), path(4), title(5), curation(6), content_hash(7), word_count(8), body(9), last_modified(10)`. The FTS5 columns `title(0), concepts(1), body(2)` would map to `documents` columns `id(0), workspace(1), type(2)` -- completely wrong! FTS would index the `id` column as "title", the `workspace` column as "concepts", and the `type` column as "body".

This is a showstopper bug in the schema. The FTS5 `content=` external content mode requires either:
(a) The FTS columns match the content table columns by ordinal, OR
(b) Manual sync via explicit `INSERT INTO fts_content(rowid, title, concepts, body) VALUES(...)` -- bypassing the content table mapping entirely.

The spec uses approach (b) in the `rebuild_workspace()` logic (Section 4.1, step 4: "insert FTS row"), which would work correctly IF the explicit inserts are always used. But the `content='documents'` declaration is still misleading because `snippet()` and other content-reading FTS5 functions will try to read from the content table by ordinal -- and get wrong data.

Also: the `documents` table has no `concepts` column. FTS5 references `concepts` but it does not exist in the content table. This will cause errors when FTS5 tries to read content for `snippet()` generation.

**Severity:** Critical

**Evidence:**
- Spec Section 4.1, lines 159-171: `documents` table schema -- no `concepts` column, `body` at ordinal 9
- Spec Section 4.1, lines 189-197: FTS5 schema with `concepts` column referencing nonexistent content table column
- Spec Section 4.6, line 559: `snippet(fts_content, 2, ...)` assumes column 2 is body, but ordinal mapping would point to the wrong column

**Recommendation:**
1. Either add a `concepts` column to the `documents` table, OR remove `concepts` from the FTS5 column list.
2. If using external content mode with manual sync (approach b), consider dropping `content='documents'` and using a regular (non-external-content) FTS5 table. This avoids the ordinal mapping confusion entirely and makes `snippet()` work correctly because FTS5 stores its own copy of the text.
3. Alternatively, if keeping external content mode, reorder the FTS5 columns to match the `documents` table ordinals exactly, or use a contentless FTS5 table (`content=''`) which never reads from a content table.

### Additional Attack C: FTS5 Query Injection

**Attack description:** What happens with `'); DROP TABLE documents;--` as a search query?

**Finding:** The spec uses parameterized queries (`WHERE fts_content MATCH ?` with `params = [safe_query]`). Parameterized queries prevent SQL injection. The `_sanitize_fts_query()` function (Section 4.6, step 3) escapes user input for FTS5 syntax, and `OperationalError` from malformed queries is caught.

However, the sanitization logic is underspecified. The spec says "Escape `\"` characters" and "if query contains FTS5 operators, pass through." This means a user can craft FTS5 prefix queries like `"title":architecture*` to search only the title column. This is probably desirable (power user feature), but could confuse LLMs that construct queries. More importantly, queries like `{title concepts body}:*` would match all documents -- a potential DoS for large portfolios (though at 400 docs this is irrelevant).

The real concern is that `NEAR(term1 term2, distance)` syntax passed through could produce unexpected result counts. The spec should test edge cases like empty string (already handled), single-character queries, and very long queries.

**Severity:** Minor (injection prevented; edge cases remain)

**Recommendation:** Add test cases for FTS5 operator pass-through: `AND`, `OR`, `NOT`, `NEAR`, column filters, prefix queries. Document that FTS5 query syntax is intentionally exposed.

The answer matters because `project_registry()` returns whatever the scanner puts in the DB. A ghost project with `doc_count: 0` and `status: "undocumented"` would confuse LLMs.

**Severity:** Minor

**Recommendation:** Explicitly specify: if a registry project's path does not exist on disk, skip it entirely (do not insert into `projects` table). Log a warning to stderr.

### Additional Attack E: `promote_document()` Returns `success: false` Without `isError: true`

**Attack description:** Per Section 4.8.4, `promote_document()` returns `{success: false, validation_warnings: [...]}` when L2 requirements are not met. Per the MCP protocol, `isError: true` signals tool failure to the LLM. The spec returns a normal (non-error) response with `success: false`. How does the LLM distinguish "tool worked, promotion denied" from "tool succeeded, promotion completed"?

**Finding:** The `_invoke_tool()` wrapper (server.py lines 271-293) returns `CallToolResult(isError=False)` for all non-exception returns. The `PromoteDocumentResponse` schema uses `success: bool` (not `Literal[True]`). So when promotion validation fails, the response is `CallToolResult(isError=False, structuredContent={success: false, validation_warnings: [...]})`.

From the LLM's perspective, `isError: false` means "tool executed successfully." The LLM must inspect the `success` field to know the promotion was denied. This is a reasonable design (the tool DID execute successfully -- it checked validation and correctly reported the result). But it is inconsistent with the write tool shared property stated in Section 4.8: "Errors returned via `isError: true` with (what happened, why, how to fix)."

Validation failure for `promote_document()` is arguably an error (the requested action was not performed), not a success. The research document (Area 2) says: "Tool errors returned with `isError: true` go into the LLM context for self-correction." If promote validation failures are returned without `isError: true`, the LLM has less context for self-correction.

**Severity:** Minor

**Recommendation:** Clarify in Section 4.8.4: validation failure should return `isError: true` with `E_PROMOTION_BLOCKED` error code containing the validation warnings. The `success: false` + `isError: false` pattern creates ambiguity. Use `isError: true` consistently for all cases where the requested action was not performed.

## 7. Risk Assessment Override

### Track A: MEDIUM -- AGREE

The SQLite/FTS5 layer is new but well-understood technology. The schema inconsistency (Attack A) is a specification bug, not an architectural risk -- it is fixable in implementation. Read-only surface limits blast radius. The FTS5 schema bug is the only critical issue; everything else is minor.

If OQ-2 is resolved as (B), Track B risk increases to HIGH because file rename is new, untested code.

### Shared: LOW -- AGREE

Registration plumbing and schema additions are mechanical. The `workspace_id` threading has the logic error in `get_context_bundle()` (Attack 5), but that is a localized fix.

## 8. Blind Spot Identification

### 1. The spec conflates "wraps existing CLI" with "calls existing CLI functions"

The spec repeatedly says tools "wrap" CLI commands (scaffold, rename, log, promote). But "wrapping" can mean:
- (a) Calling the CLI command's internal functions directly
- (b) Calling `subprocess.run(["ontos", "rename", ...])` 
- (c) Re-implementing the same logic in a new module

The spec uses (a) for `log_session` (calls `create_session_log()` directly), but for `rename_document` it says "adapt the rename command's internal logic" and "use `_compute_rename_plan()` (if exposed) or re-implement." The `_compute_rename_plan` function does not exist -- there is `_prepare_plan()`. This ambiguity means an implementer might re-implement rename logic from scratch rather than reusing the tested code, introducing new bugs.

### 2. No consideration of MCP tool timeout

Stdio MCP sessions have no protocol-level timeout, but MCP clients (Claude Code, Cursor) may impose their own. `rename_document()` on a workspace with 400 documents could take several seconds (scan all documents, build plans, write files, refresh cache, rebuild portfolio index). The spec does not mention performance targets or timeout considerations for write tools.

### 3. No error recovery guidance for end users

The spec describes `isError: true` responses with "what, why, how to fix" -- but all error recovery guidance targets the LLM agent, not the human user. If the LLM gets stuck in a loop (e.g., retrying a locked workspace), there is no escalation path documented. The `E_WORKSPACE_BUSY` error says "Retry in a few seconds, or delete .ontos.lock if the process is gone" -- the "delete .ontos.lock" part is for the human, but the LLM might interpret it as an instruction to call a filesystem tool.

### 4. `documents` table lacks a `concepts` column

The FTS5 schema references a `concepts` column that does not exist in the `documents` table. This is related to Attack A but worth calling out as a blind spot: the spec author apparently carried the `concepts` column from the FTS5 design without verifying it exists in the content table. The concept of "concepts" as a searchable field needs a data source -- where do concepts come from? From frontmatter `concepts:` field? If so, this should be extracted during `rebuild_workspace()` and stored in `documents.concepts`.

### 5. The spec does not address what happens when `mcp >= 1.27.0` is not available

Section 3 (Dependencies) says "if `mcp>=1.27.0` is not yet released when implementation begins, pin to the latest available `>=1.2`." But the spec's code samples import from `mcp.types` and use features that may differ between 1.2 and 1.27.0. No compatibility layer or conditional import is described.

## 10. Verdict

**Request Changes**

The spec is architecturally sound and impressively detailed for a v1.0 draft. The 1,776 lines cover every major design decision with code-level specificity. However, three blocking issues must be resolved before implementation:

(1) The FTS5 schema is internally inconsistent and will produce wrong results at runtime -- the `concepts` column does not exist in the `documents` table, the two schema definitions contradict each other, and the external content ordinal mapping is broken. This is a Critical bug that requires a single canonical schema.

(2) The spec claims the existing `ontos rename` CLI already implements file renaming, but it does not. This either means OQ-2 should be resolved as (A) ID-only (matching reality), or file rename must be designed as genuinely new scope with its own edge cases.

(3) The atomicity claims for `SessionContext.commit()` are overstated, and the dual lock file gap between CLI and MCP creates a real concurrent-write risk. These must be documented honestly even if not fully fixed in v4.1.

(4) The `get_context_bundle()` logic error when `workspace_id` is passed in single-workspace mode is a straightforward code bug in the spec.

All four issues have clear fixes. None requires rearchitecting. I estimate 2-3 hours of spec revision to address all blocking issues and the major items. The spec is 85% ready.

**Confidence level:** High. All findings are backed by direct codebase evidence with file paths and line numbers. I read every line of the five required codebase files and both source documents.
