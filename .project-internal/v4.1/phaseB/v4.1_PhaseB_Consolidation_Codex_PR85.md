## Agreement Analysis

- **Strong agreement**
  - All three reviewers independently landed on the FTS/schema/content-sync issue as a real blocker, not a wording problem.
  - All three reviewers found the single-workspace vs portfolio contract unresolved, especially where `workspace_id`, `refresh()`, and `--read-only` interact with backward-compatibility claims.
  - All three reviewers found the write-safety story overstated relative to the current codebase, particularly around lock authority and rename semantics.

- **Single-reviewer or lightly corroborated findings worth preserving**
  - R1 surfaced the missing typed write-error schema and the overly optimistic write-tool annotations.
  - R2 surfaced the proposal D2 mismatch for existing-tool `workspace_id` behavior and the fact that input-schema changes cannot be solved by editing output Pydantic models alone.
  - R3 surfaced slug-collision risk, partial-workspace snapshot failure, `event_type=\"session\"` ambiguity, multi-process portfolio DB contention, and bundle determinism problems.

- **Direct disagreements**
  - **OQ-1 (`get_context_bundle()` in single-workspace mode):** R1 still prefers a tightened version of recommendation B. R2 and R3 prefer A to preserve the current 8-tool single-workspace contract and collapse the compatibility ambiguity.
  - **OQ-2 (path rename vs ID-only):** R1 prefers B-with-guardrails. R2 rejects default full file rename unless spec and code explicitly prove parity with the current planner/apply path. R3 prefers C with a conservative default of ID-only.
  - **Blocker ordering:** R1 puts FTS/schema first, R2 puts tool-surface clarity first, and R3 puts false atomicity first. This is a priority difference, not a contradiction about the underlying facts.

- **Factual disagreements vs priority disagreements**
  - **Factual issues that need resolution in the spec:** whether `documents` actually persists every FTS-weighted column; whether existing tools truly accept optional `workspace_id` in portfolio mode; whether the spec expects absent write tools or tool-level write rejections in `--read-only`; and whether one or two lock authorities exist in the actual write path today.
  - **Priority disagreements left to CA judgment once the facts are corrected:** the exact blocker ordering, how aggressively to call out risk inflation, and whether the OQ-2 default should be conservative by configuration or conservative by fixed behavior.

## Required Actions for CA

1. **Fix the Track A data contract first** (`Medium`): rewrite 4.1, 4.6, and OQ-3 so the persisted schema, FTS table, snippet indexing, and workspace-local rebuild/delete/update logic are internally consistent and executable.
2. **Resolve the MCP surface contract second** (`Medium`): decide the exact tool matrix for `ontos serve` vs `ontos serve --portfolio`, then update 4.4, 4.7, 4.8, 4.10, 6, 7, and 10 to remove the current contradictions.
3. **Rewrite the Track B safety contract third** (`Medium-Large`): unify lock authority with the current write path, choose the real atomicity contract, and revise OQ-2 so the default behavior matches what the current implementation base can safely support.
4. **Define the write-error envelope** (`Small-Medium`): add a typed error schema and settle the exact `--read-only` observable behavior. The current spec should not require both “tool absent” and “tool error” at the same time.
5. **Re-baseline risk, tests, and diagrams** (`Small-Medium`): once the contracts are fixed, rerate Track A / Track B / Shared, expand the tests around the corrected decisions, and update diagrams to match the final prose.

**Consolidated position on OQ-1:** **Choose (A) for v4.1**. Keeping `get_context_bundle()` portfolio-only is the cleaner fix because it preserves the current single-workspace MCP surface and removes a large share of the compatibility and diagram/prose ambiguity. If the CA still wants (B), the spec must explicitly accept the surface change and rewrite the backward-compatibility claims accordingly.

**Consolidated position on OQ-2:** **Choose (C), configurable, with a conservative default of ID-only behavior**. The current reviewer evidence does not justify implicit full path rename as the default contract. If file rename is offered, the spec should require explicit reuse of the existing planner/apply path or equivalent proof that path-rename parity is safe.

## Decision Summary

**Decision:** `Needs Revision`

The release scope does **not** need re-scoping yet. The direction remains aligned with the re-scoped v4.1 proposal: stdio-only, portfolio index, flat tool surface, and write tools as the main value add. What is missing is a decision-complete implementation contract for the data model, tool surface, and write-safety behavior.

The fastest path forward is to revise the spec rather than restart it. Once B1-B3 are resolved, the remaining issues look like normal tightening work rather than a direction-level failure.
