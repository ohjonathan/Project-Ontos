| Spec claim (v1.1) | Drift discovered | Addendum ruling | Shipped behavior |

## 2.A Track A implementation contracts (spec v1.1 → post-PR-85 reality)

Implementation-level drift captured during Track A's Phase D fix cycle. Each row became a contract Track B had to honor.

| Item | Spec v1.1 says | Codebase reality post-PR-85 | Why it changed | Impact on Track B |
|------|----------------|-----------------------------|----------------|-------------------|
| **Workspace slugify** | Implementation per subsystem | Single shared function at `ontos/mcp/scanner.py:35`; `server.py` + `tools.py` delegate; `verify.py` uses scanner helpers | CB-2/CB-6: three divergent slugify impls were root cause of verify false positives. Consolidated. | **Hard contract.** Track B write tools that create/modify workspaces MUST import the shared slugify. No new fallback implementations. |
| **Config validation** | Strict — unknown keys raise `ConfigError` | Strict with two named exceptions: `validation.strict → hooks.strict` legacy mapping, and top-level `[project]` section pass-through | CB-1 + D.5 amendment. `[project]` is legitimate; raising would be a regression. Both exceptions targeted, not blanket silent-drop. | Track B must handle `ConfigError` from strict validation. New config sections must be added to the allowlist explicitly. |
| **`has_ontos` detection in verify** | Context-map field inference | Filesystem fallback to `.ontos.toml` presence at `verify.py:444`, with tightened boolean coercion (`"false"` not truthy) | Phase C organic addition; accepted via §14 Option A in Phase D | Track B verify-adjacent tools reuse the fallback. |
| **Registry parsing** | Verify + scanner had separate impls | Shared parser `scanner.load_registry_records` at `ontos/mcp/scanner.py:143` accepts `projects` / `workspaces` / `entries` / `items` / dict-shaped registries | CB-4 | **Hard contract.** Track B must use the shared parser. No parallel registry parsing. |
| **`SessionContext.commit()` semantics** | "Atomic writes" (old docstring) | Buffered commit, NOT atomic. Docstring updated. | SF-7: codebase never provided true multi-file atomicity | Track B designs for partial-commit detection, not rollback assumptions. |
| **`_reset_db_file()` corruption recovery** | Delete main DB file | Delete main DB + `-wal` + `-shm` sidecars | CB-7: SQLite was replaying stale WAL against new DB | Track B write operations that modify the portfolio DB handle sidecar cleanup on corruption. |
| **`fcntl.flock` descriptor handling** | Standard `LOCK_EX \| LOCK_NB` | Plus `os.set_inheritable(fd, False)` at `locking.py:22` | CB-8: lock leaked to child processes via `subprocess.Popen(close_fds=False)` | Track B write tools using `workspace_lock()` inherit this for free. |
| **FTS5 concurrent rebuild** | `BEGIN IMMEDIATE` for rebuild | Plus SQLite `timeout=5.0`; busy `OperationalError` mapped to `E_PORTFOLIO_BUSY` semantic error | CB-10: concurrent reads failed with raw `OperationalError` | Track B write tools handle `E_PORTFOLIO_BUSY` as retryable. |
| **Portfolio tool routing errors** | Generic exception | Structured `isError: true` payload for missing portfolio tool invocations | CB-9 | Track B write tools follow the same `isError: true` pattern. |
| **`workspace_id` propagation** | Implicit | Threaded explicitly through `_cmd_verify` in portfolio mode | CB-11 | Track B write tools accept and propagate `workspace_id` explicitly. |
| **`requires-python`** | `>=3.9` | `>=3.9` (briefly bumped to `>=3.10` in Phase C, reverted) | CB-5: `mcp` extra's own dep naturally prevents 3.9 from installing `ontos[mcp]` — no need to bump base | Track B did not bump `requires-python`. |

## 2.B Track B implementation contracts (addendum v1.2 → post-PR-92 reality)

Track B's own drift — addendum promises vs. shipped codebase.

| Item | Addendum v1.2 says | Codebase reality post-PR-92 | Why it changed | Impact forward |
|------|--------------------|------------------------------|----------------|----------------|
| **`workspace_id` contract** | Required; missing returns structured error (original Track B design §182, §203) | Optional; missing falls back to served workspace | SF-B6: implementation and user-facing manual were correct; design doc was stale. Aligned doc to code. | Future write tools follow the same contract. Single-workspace MCP sessions don't need `workspace_id`. |
| **Rename scope** | Scope-tracks-config (Dev 3 refactor accepted ruling) | Unconditional `ScanScope.LIBRARY` at `rename_tool.py:429` | CB-B3 reversal: scope-tracks-config was cross-workspace lock bypass under `default_scope = portfolio`. Earlier acceptance was wrong. | MCP rename never triggers cross-workspace body rewrites. Any future rename-semantics expansion needs explicit multi-workspace locking. |
| **Single-file write tool error path** | A3: rollback-then-rebuild | Rollback-then-rebuild via `rollback_path()` helper in `ontos/core/git.py` | SF-B7: implementation aligned to addendum. Initial impl did rebuild-on-error only. | A3 is now universally honored across write tools. Pattern: tracked files → `git checkout --`, untracked → `unlink`. |
| **`read_only` server registration** | Write tools absent from `list_tools()` when `read_only=True` | Same — `_register_write_tools()` guarded by `if not read_only` at `server.py:164` | CB-B2 | Future write tools inherit the guard automatically. |
| **Rebuild slug derivation** | Canonical slug via `allocate_slug` (collision suffixes) | Canonical slug via `resolve_workspace_slug` at `ontos/mcp/_validation.py:53` | CB-B4: original impl used `workspace_root.name` → `sqlite3.IntegrityError` under basename collisions | Future write operations that rebuild portfolio index use the canonical slug helper. No exceptions. |
| **Bundle config wiring** | Three fields consumed by `build_context_bundle()` with defaults `8000 / 20 / 30` | All three fields wired at `tools.py:495-502`; defaults applied | CB-B1: initial Dev 4 work wired `token_budget` but not `max_logs` / `log_window_days` | Future bundler changes thread all three fields through the public API. |
| **Rename plan builder** | Shared `build_rename_plan` between CLI and MCP | Shared builder at `ontos/commands/rename.py`; MCP routes through it via injected `project_root` parameter | Dev 3 refactor eliminated the parallel per-caller loop. CWD-vs-injected equivalence test added. | Future rename-semantics changes affect CLI and MCP uniformly. No second implementation tolerated. |
| **`_validate_workspace_id` typing** | `Optional[PortfolioIndexLike]` + `Optional[str]` | Same at `ontos/mcp/tools.py:524` (delegation body) and `ontos/mcp/_validation.py:85` (shared helper) | CB-B5 + m-B1: stacked-merge conflict resolution (Dev 2 body + Dev 4 type hint) | Type contract is explicit. Callers don't default to `Any`. |
| **Python 3.14 FrozenInstanceError** | Not addressed in spec | Catch-inside-`with` workaround at `ontos/mcp/writes.py:306` and `ontos/mcp/rename_tool.py:334`; error-code preservation in envelope helpers | Python 3.14 `@contextmanager` unwinding attempts `exc.__traceback__ = ...` on a frozen dataclass → crash. Pragmatic workaround; unfreeze follow-up deferred. | **Deferred to v4.1.2:** unfreeze `OntosUserError`/`OntosInternalError` to eliminate the workaround. |
| **Bundler sort surfaces** | Deterministic ordering | Six defensive sort surfaces in `bundler.py` (lines 55, 140, 159, 166, 193, 206, 271) | SF-B2 architectural finding: redundant defensive sorts mean any single-sort deletion is non-impactful. Regression test covers collective determinism across three dict-order permutations. | **Future refactor consideration:** consolidate sort surfaces. Not blocking. |
| **Write tool documentation** | Documented in `Ontos_Manual.md` | Four write tools documented at `Ontos_Manual.md:700+` with schemas and `read_only` / `workspace_id` behavior | SF-B4 | User-facing contract matches implementation. |

### 4.4 Architectural findings surfaced but NOT addressed this cycle

1. **SF-B2 over-strict requirement.** Directive said "deleting any single sorted() call must break the test." Empirically impossible: bundler has six redundant defensive sorts (lines 55, 140, 159, 166, 193, 206, 271), and removing any one leaves the others sufficient for determinism. Test strengthened to three permutations (strictly stronger than pre-fix single reversal) but NOT to per-sort sensitivity. Flagged to orchestrator in PR #92 comment. Follow-up recommendation: either collapse the defense-in-depth (risky — removes redundancy) or accept the collective-determinism test.
2. **Verifier error in fix cycle 1.** D.5 verifier claimed CB-B4 and CB-B5 production fixes landed; HEAD verification proved otherwise. The re-fix cycle 2 closed them. No silent skipping — discrepancy flagged in PR #92 comment and re-fix directive from orchestrator acknowledged the error.
