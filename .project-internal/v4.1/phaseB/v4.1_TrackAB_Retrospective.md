# Ontos v4.1 Track A/B Retrospective

**Cycle:** v4.1 "Portfolio Authority"
**Duration:** 2026-04-11 → 2026-04-12 (compressed cycle; normally this would span weeks)
**Shipped:** v4.1.0 at commit `9254064`
**Author (CA):** Claude Opus 4.6

This retrospective captures what worked, what broke, and what future Ontos cycles should carry forward. The goal is not blame or ceremony — it's durable guidance that makes the next cycle cheaper and more reliable.

---

## 1. What worked

### 1.1 The multi-platform review model

Track A Phase D used 9 reviewers across 3 model families (Claude Code, Codex, Gemini). The overlap and divergence pattern produced a much richer verdict than any single platform could:

- Codex caught the dead-bundle config (SF-3) that Claude Code missed.
- Gemini caught the `fcntl.flock` FD inheritance leak (CB-8) that neither other platform flagged.
- Claude Code caught the three divergent slugify implementations (CB-6) via structural analysis.
- Each platform independently reached "Needs Fixes" verdict; cross-validation prevented a single-reviewer false-positive from dominating.

**Carry forward:** multi-platform review is non-negotiable for substrate PRs. The single-platform findings were often the most valuable ones — never downgrade a finding because only one platform caught it.

### 1.2 C.0 investigation before Phase C

Track B's C.0 investigation (before writing any code) caught the double-lock deadlock as a SPEC defect, not an implementation bug. Discovering it pre-code saved a full Dev-1-redesign-loop.

**Carry forward:** any Phase C implementation that depends on a substrate feature (locks, slugifiers, shared parsers) gets a C.0 pass first — read the substrate code, identify contract gaps, write an addendum BEFORE dev-agent spawns.

### 1.3 Parallelizing with isolated worktrees

Devs 1 and 4 ran in parallel; Devs 2 and 3 ran sequentially on Dev 1's branch tip. Four parallel worktrees × isolated branches enabled concurrency without shared state. Merge simulation before actual consolidation confirmed conflict surface BEFORE it hit the integration branch.

**Carry forward:** for any track with independent slices, pre-merge simulation on a throwaway branch is the cheapest way to detect conflicts. Do it before opening the final PR, not after review.

### 1.4 Addendum-as-authority pattern

Addendum v1.2 superseded specific sections of v1.1 rather than rewriting the spec. This preserved review traceability (v1.1 review debt was still valid for unchanged sections) and avoided the "which spec is current?" confusion.

**Carry forward:** when spec drifts during implementation, addendum a narrow supersession instead of publishing v1.2-full. The drift table in the sequence ledger is the diff.

## 2. What broke

### 2.1 Verifier drift: stale HEAD reads twice

D.5 verifier claimed CB-B4 and CB-B5 production fixes had landed. HEAD verification proved otherwise — both were unlanded despite claims. The same class of error hit again on the re-fix cycle (CB-B1 flagged as live when it had actually been wired by Dev 4, just not regression-tested).

**Root cause:** verifier agents read stale local state (their worktrees were behind origin). They trusted the remote's claim rather than pulling + re-verifying.

**Carry forward:**
- Every verifier checks `git log -1 --oneline` on origin FIRST, pulls to a clean state, re-runs full test suite, THEN verifies specific lines.
- Every verdict cites the exact commit hash it was verified against.
- Re-fix directives should be signed against a specific commit hash, not a branch name.

### 2.2 Scope-tracks-config accepted in error (CB-B3)

Dev 3's shared `build_rename_plan` orchestrator refactor introduced `scope = resolve_scan_scope(None, config.scanning.default_scope)` as a side-effect — well-intentioned "preserve MCP behaviour" glue that actually leaked a CLI-surface preference into MCP write tools. The CA (me) approved it during the refactor review. Only surfaced as CB-B3 in Phase D.

**Root cause:** the refactor's stated goal was "unify CLI and MCP plan building." Honoring workspace `default_scope` from MCP was subtly MORE unification than the directive required, and I didn't catch the over-reach.

**Carry forward:**
- Refactors that "unify" call paths MUST explicitly preserve each caller's surface contract. Any NEW behavior borrowed from the other caller is spec-expanding and needs spec-level approval.
- When a refactor affects multiple entry points, the PR description should list per-entry-point behavior diffs, not just "unified."

### 2.3 Single-platform fix-cycle led to regression

The initial Track B fix cycle was a single-dev-agent-per-cycle model (me as CA, Dev agents executing). When Dev agents got blocked by harness plan-mode propagation, I executed directly. That was the right call for the blocker, but it collapsed the CA/Dev role separation that the cycle relied on. The result: my own self-approved fixes didn't always have the outside-perspective quality check that multi-agent review provides.

**Carry forward:**
- CA-executes-directly is a valid escape hatch but should be logged as a process exception. The escape hatch cost for v4.1 was: two P0s (CB-B2, CB-B3) and two SFs needed a re-fix cycle because the initial cycle never actually ran.
- If Dev agent spawns fail repeatedly, investigate the HARNESS issue before pivoting to direct execution — the root cause is usually recoverable with a one-line prompt change (e.g., explicit `ExitPlanMode` at spawn-time).

### 2.4 Sequence ledger was missing throughout v4.1

C.0 findings explicitly noted: "A `v4.1_Sequence_Ledger.md` does not exist in this repo." Each stage produced its own meta-doc but there was no canonical rollup until this retrospective was written.

**Carry forward:**
- The sequence ledger is the v4.x-wide source of truth. Start it at proposal approval, not at release. Update after every stage boundary.
- Treat the ledger as a required deliverable, not a nice-to-have.

### 2.5 Test count as process signal

The directive "test count must strictly increase" caught a real issue in the first fix cycle (1113 → 1113 proved nothing landed) but also created pressure to add trivially-passing tests to inflate the number. The re-fix cycle's directive refined this to "strictly increase OR explain why" — better, but still imperfect.

**Carry forward:** test count is a sanity check, not a proof of correctness. Pair it with: (a) regression tests must fail against pre-fix code — verified empirically, not claimed; (b) grep-verifications for the exact code change.

## 3. Architectural findings deferred

### 3.1 Bundler defense-in-depth vs per-sort sensitivity

`bundler.py` has six redundant defensive sorts. Removing any one is survivable (determinism preserved by others). SF-B2's directive to make a per-sort-removal test break is architecturally unachievable without first flattening the defense.

**Option A:** flatten the defense (risky — removes redundancy).
**Option B:** accept collective-determinism testing (shipped).
**Option C:** instrument the sorts with pytest fixtures that selectively patch them out during CI (adds complexity).

Decision deferred to v4.2 scope.

### 3.2 Python 3.14 `FrozenInstanceError` on `OntosUserError`

Frozen dataclass + `@contextmanager` unwinding = crash on exception traceback assignment. Dev 2 worked around it by catching inside the `with workspace_lock()` block; Dev 3 matched the pattern. SF-B3 pinned the workaround with a regression test.

**Real fix:** unfreeze `OntosUserError` and `OntosInternalError`. Out of scope for v4.1.

### 3.3 `_rebuild_safely` is duplicated

Two implementations — `writes.py::_rebuild_safely` (simple swallow) and `rename_tool.py::_rebuild_safely` (with m-13 retry). Different contracts justified the split, but the duplication is lint-bait for future developers.

**Follow-up:** unify via a `ontos/mcp/_recovery.py` module with retry-policy parameter.

## 4. Process meta

### 4.1 Meta-prompt sprawl

The v4.1 cycle accumulated prompts at every stage: proposal, spec review R1/R2/R3, B.4 verification, Phase B consolidation (3 platforms), C.0 findings, dev team meta-prompt, Phase D verdict, fix cycle, re-fix, final re-fix, merge/release directive. Each was a different document. The CA burden was "re-read the full stack of prompts before every action."

**Carry forward:** the sequence ledger IS the meta-prompt index. Future cycles should be able to ask "what did stage X conclude?" from one doc, not by scavenging across 15 review files.

### 4.2 Merge-simulation is cheap insurance

Before opening the v4.1 consolidation PR, the CA ran a local 4-PR merge into a throwaway branch and ran the test suite. This caught the Dev 2/Dev 4 `tools.py::_validate_workspace_id` conflict BEFORE it hit GitHub. The resolution ("Dev 2's delegation body + Dev 4's type hint") was obvious in isolation and scripted before the real merge.

**Cost:** ~15 minutes of CA time. **Value:** one less GitHub review cycle.

**Carry forward:** mandatory pre-consolidation simulation for any track with parallel dev slices.

### 4.3 The re-fix cycle isn't free

v4.1 Track B took 2 revision cycles (plus one "final re-fix") to close. That's within spec (one revision cycle is expected per §2.2 of the verdict protocol) but suggests the initial fix cycle was under-resourced on verification. Specifically: the initial fix cycle's own test-count delta was 0 — a process violation that WAS caught by D.5, but only after a full re-review round.

**Carry forward:** every fix-cycle commit includes a regression test in the SAME commit, and the CA runs full tests before accepting the cycle as done. D.5 should be a formality, not a discovery phase.
