## Compatibility analysis

The `frozen=True` → `unsafe_hash=True` + `_FieldImmutableException` rework is the most delicate compatibility move in the diff, and it lands correctly. The public field-immutability and hash contract from v4.7.0 is preserved (assignment to `message`/`code`/`details` still raises `FrozenInstanceError`), while `BaseException`-owned metadata (`__traceback__`, `__cause__`, `__context__`, `__suppress_context__`, `__notes__`) is now assignable. This is a genuine correctness fix, not a contract break: under Python 3.14, `contextlib` assigns `__traceback__` when propagating through a generator context manager, and a frozen dataclass would have swapped the real error for a spurious `FrozenInstanceError`. The allowlist is narrow enough that it cannot be used to mutate the public fields.

The intentional behavior changes are all in-scope and documented in CHANGELOG §Compatibility (lines 34, 50-53): log-slug collisions now error instead of overwriting, unsafe mutation paths fail before touching external data, and mutation of invalid UTF-8 refuses with exit 1 / `E_COMMAND_FAILED` while read-only loading keeps replacement decoding.

One consequence deserves an explicit callout that I did not find in §7. A document carrying a YAML-coerced non-string ID — `id: 2026-07-10` (date) or `id: 123` (int) — previously loaded and now drops out of the canonical document set with a `parse_error` issue. This is squarely within the spec's scope bullet ("Reject non-string or malformed document IDs before a document is written or admitted to the canonical loader") and is the correct fix, but it is a visible change to the loadable document set in a patch release, and §7's "Existing valid documents … keep their format and envelope contracts" reads as if nothing disappears. Such documents were never *valid*, so the sentence is technically true — I'd still add one line to §7/CHANGELOG naming the symptom operators will actually see, because the failure mode is "my doc vanished from the map," not "my doc was rejected."

`tomli_w` is correctly declared as a hard runtime dependency at baseline (`pyproject.toml` dependencies, unchanged by this hotfix — the only pyproject edit is the 4.7.0 → 4.7.1 bump). Installed users are unaffected by Issue 1.

## Issues

### Issue 1 — Full suite does not pass; acceptance criterion 2 fails (blocking, test-only)

Three tests fail in the current working tree:

```
FAILED tests/mcp/test_error_paths.py::test_help_and_stdout_safety_paths
FAILED tests/mcp/test_error_paths.py::test_json_serve_startup_error_stays_on_stdout
FAILED tests/mcp/test_parity.py::test_count_parity_and_cli_loader_parity
3 failed, 1557 passed, 2 skipped in 62.78s
```

Reproduction:

```
$ .venv/bin/python -m pytest tests/mcp/test_error_paths.py tests/mcp/test_parity.py -q
...
E   assert 1 == 0
E    +  where 1 = CompletedProcess(args=['python3', '-m', 'ontos', 'serve', '--help'],
     returncode=1, stderr="...ontos/io/toml.py", line 12, in <module>
     import tomli_w
     ModuleNotFoundError: No module named 'tomli_w'")
```

Single root cause. `tests/mcp/__init__.py:302` spawns the CLI as `["python3", "-m", "ontos", ...]` — a bare interpreter resolved from `PATH` (here Homebrew's system Python 3.14), not `sys.executable`, so the subprocess runs outside the venv where Ontos's dependencies live. This latent non-hermeticity was harmless at `bf91b42` and is newly surfaced by this hotfix: `ontos/io/toml.py` now imports `tomli_w` at module top level (line 12, added by this diff when `write_config` was rewritten to emit standards-compliant TOML). Any `ontos` subprocess launched under an interpreter without `tomli_w` now dies at import.

This is a test defect, not a product defect — `tomli_w>=1.2,<2.0` is a declared runtime dependency, so a real install is fine. But spec §6 AC 2 states "The complete suite passes from the hotfix environment," and it currently does not. That gate is failing on its own terms, and §7 of the manifest cannot honestly report green.

Fix is mechanical: replace `"python3"` with `sys.executable` in `run_base_cli`/`run_mcp_cli` (`tests/mcp/__init__.py:298-315`). Two other files hardcode the same string (`tests/test_doctor_mcp_probe_regression.py`, `tests/commands/test_consolidate_parity.py`) and should be swept in the same pass before they fail for the same reason.

### Issue 2 — Legacy `LOGS_DIR` discovery narrowed from import-path to two fixed locations (non-blocking, needs a §7 line)

Baseline resolved the legacy config via `from ontos_config import LOGS_DIR`, i.e. anything importable on `sys.path`. `_load_legacy_logs_dir` now probes exactly `<root>/ontos_config.py` and `<root>/.ontos/scripts/ontos_config.py`. Removing the CWD/`sys.path` dependence is the right call and the module-state save/restore around `ontos_config_defaults` is careful work. But a project that today gets `ontos_config` from an installed package or a `PYTHONPATH` entry will silently fall through to `[paths].logs_dir` and begin writing logs to a different directory. Spec §7 claims legacy precedence is "retained"; it is retained only for project-local configs. Worth one sentence in §7 naming the narrowed discovery, since the failure is silent and relocates data.

## Positives

The contract-first split is the strongest part of this deliverable and it holds up under direct verification. All ten §9 exclusion paths — including `ontos/cli.py`, `ontos/ui/json_output.py`, `ontos/core/graph.py`, and `ontos/mcp/schemas.py` — are byte-identical to baseline, and `json_output.py` is byte-identical to `main`, so the schema-3.4 envelope claim is mechanically true rather than asserted. The maintainer resisted the obvious temptation to let the serializer rewrite pull `graph.py` or the MCP schemas along with it.

The explicit-ID-versus-fallback asymmetry is exactly right and is the subtlest judgment call in the patch. Applying the ID grammar to filename stems would have silently deleted `My Notes.md` and `café-doc.md` from the canonical loader; the code comment at `ontos/io/files.py:449-452` states that reasoning, and the parametrized fallback test pins it. Ordering the README/template exclusion *ahead* of the validator so `_template` stems never reach the grammar shows the author traced the interaction rather than bolting validation onto the front.

Hermeticity (AC 3) is genuinely clean: a full suite run produced zero tracked or untracked repository changes, verified by pre/post `git status --porcelain` comparison. The `_FieldImmutableException` docstring correctly identifies a real Python 3.14 `contextlib` interaction, which is the kind of root-causing that usually gets papered over with a blanket unfreeze.
