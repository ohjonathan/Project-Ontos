Review written to `docs/reviews/project-ontos-v4-7-1-hotfix/D.1-claude-peer.md`. **Verdict: Request changes** — on two reproducible Medium findings, neither of which reopens the P0 corruption class the hotfix exists to close.

**What's solid.** The high-risk core holds up. `serialize_frontmatter` delegates every scalar to `yaml.safe_dump` and refuses to return unless the text reloads to an equal value — the round-trip assertion is the contract, which is the right shape for this fix. The transaction envelope does what the spec describes: `O_EXCL` staging with unpredictable names, `lstat` throughout so links aren't followed, mode preserved via `fchmod`, `fsync` before `os.replace`, and real rollback on multi-op commit failure. MCP outer/inner locks can't diverge because the typed guard is passed into a `SessionContext` with `owns_lock=False`. v5 exclusion parity is clean (empty diff across all nine excluded files, no `command_registry.py`, no golden touched), both version sources say 4.7.1, and the full suite is green at 1563 passed / 2 skipped with no tracked repo changes.

**The two findings.**

The read-only MCP portfolio can serve indefinitely stale rows. `_connect` opens with `?mode=ro&immutable=1`, which tells SQLite to ignore the `-wal` sidecar entirely — but that immutability promise doesn't hold, since a writable process can rebuild the same database. The hotfix upgraded `rebuild_all` to a `TRUNCATE` checkpoint for exactly this reason, but `_rebuild_workspace` (the path every MCP write tool actually takes) still ends at `PASSIVE`, which no-ops whenever a reader holds the WAL. I reproduced it in a scratch tempdir: the `immutable=1` reader sees only `['before']` while a plain `mode=ro` reader sees the committed row.

The duplicate-field guard misses quoted keys. `_TOP_LEVEL_FIELD_RE` doesn't match `'id':`, so the guard sees zero occurrences and appends a *second* `id:` line. The post-write checks don't catch it because PyYAML is last-wins and `assert_frontmatter_roundtrip` compares the text against a dict parsed from that same text. In-process repro: input `'id': old-doc` plus an update to `new-doc` yields frontmatter containing both `'id': old-doc` and `id: new-doc`, while the plain-duplicate control is correctly rejected.

I withheld approval specifically on the second one: it's a direct, reproducible contradiction of a stated scope item ("rejects duplicates") on the exact surface this release hardens, reachable from `retrofit`, `rename`, and MCP `promote_document`. Both fixes are narrow and testable with patterns already in the suite.
