## 2. Feasibility

As a *single* Phase A→E deliverable, this proposal cannot become a deliverable in
a reasonable spec pass. It is a release-line program spanning four target
versions (v5.0.3, v5.1.0, v5.2.0, v6.0.0) plus a governance checkpoint, remaining
v5.x maintenance, and archive migration, across nine live issues (#149, #158,
#165, #173–#178). §21.2 enumerates roughly seventeen ultimate child scopes; the
initial scaffold alone is six. A single Phase A spec would simultaneously
specify narrow patch-level correctness work (v5.0.3 #176) and a major-version
breaking API deletion (v6.0.0 #149), which the proposal itself forbids as a
non-goal (§5: "Implementing all releases in one monolithic lifecycle
deliverable"; §26.11: "One monolithic implementation PR — Rejected").

The proposal is, however, correctly sized as a **terminal Pre-A Split parent**:
`user_facing: false`, `lifecycle_mode: framework lifecycle`, a content-only
charter whose only accepted Pre-A disposition is `Split into multiple proposals`.
`model_assignments` enables only `-A.proposal` for the parent; phases A–E are an
inert schema/P3 fallback roster that gate `G-blocker-1` prevents from starting.
Capability, resource, or authority gaps for the **parent** deliverable: none
material. The parent changes no product surface and writes no code; its sole
output is this verdict plus forward-declared child governance paths.

What would need to be true for each child to reach a spec pass: each child
re-enters `-A.proposal` with its own proposal and manifest and receives its own
non-author Pre-A decision before any Phase A work (confirmed by the tracker and
§21.4). That is downstream of this verdict and not in scope for this review.

## 4. Scope sanity

This proposal is not sized for a single deliverable. It is correctly framed as a
terminal Pre-A Split parent: it changes no product surface and its only
authorized Pre-A disposition is `Split into multiple proposals`. The initial
split has six independently reversible child boundaries, each named concretely
and matching the manifest's `G-cardinality-4` roster and the proposal's §21.2 /
§31 recommendation:

1. **board hygiene** (`project-ontos-v5-remediation-board-hygiene`) — #149/#158
   title/label/milestone/body corrections, residual-issue splits, and custody
   records. Governance only; no product code; reversible by board changes.
2. **issue #165 audit registry** (`project-ontos-issue-165-audit-registry`) —
   canonical registry, validator, renderer, and governance docs/tests rebuilt
   from the off-main prototype against current main. Blocks v5.1 release
   authority but is independent of the v5.0.3 code children.
3. **v5.0.3 dependency resolver** (`project-ontos-v5-0-3-dependency-resolver`)
   — #176 bare root-file dependency resolution through the shared graph
   resolver, with the §11.2 resolution order and ambiguous/collision/containment
   fail-closed contract. Depends only on the existing containment/allowlist
   layer.
4. **v5.0.3 resolved log paths and #149 wrapper delegation**
   (`project-ontos-v5-0-3-log-paths`) — one `ResolvedLogPaths`-equivalent
   contract consumed by all writers/readers/counters/consolidators/MCP, plus
   delegation of the seven log-related #149 wrappers to the shared resolver/query
   services while their v5 compatibility signatures stay pinned. Reverts as one
   intentional unit; independent of children 3/5/6.
5. **v5.0.3 built-in status alias** (`project-ontos-v5-0-3-built-in-status-alias`)
   — exactly one safe mapping `in-progress -> in_progress`, plan-first and
   formatting-preserving. Self-contained slice of #178; #178 stays open.
6. **v5.0.3 `[ontos].required_version` preflight**
   (`project-ontos-v5-0-3-required-version-preflight`) — bounded raw read of
   `[ontos].required_version` before strict config validation, owning only
   #178's downgrade-safety migration slice. Closes the prior ownership gap and is
   explicitly forbidden from parsing or accepting v5.1-only sections or weakening
   strict validation. Independent of children 3/4/5.

These six cross distinct boundaries (different releases, modules, review risk
profiles, and reversibility surfaces) and should be separate deliverables. The
v5.1, v5.2, remaining-v5.x, and v6.0.0 children enumerated in §21.2 are
deliberately not in the initial six; scaffolding them before v5.0.3 and #165 land
would contradict the §9 safety dependency graph. That sequencing is sound and
not a scope defect.

## 7. Verdict

Split into multiple proposals.

The proposal is directionally sound and evidence-backed (PR-P-2, PR-P-3, PR-T-1,
PR-T-2): real reproduced defects, a compatibility-respecting release boundary
(complete #178 in v5 additively; reserve v6 for the eleven promised #149
deletions), and reasonable rejection of simpler alternatives. But it is far too
broad for one Phase A→E deliverable (PR-P-1): a single spec would mix v5.0.3
patch correctness with a v6.0.0 breaking API deletion, contradicting the
proposal's own §5 non-goal and §26.11. The initial split has six independently
reversible boundaries (PR-T-4, PR-T-5, PR-T-6), each its own Pre-A child;
notably, the sixth child closes the prior ownership gap for the
`required_version` preflight without parsing or accepting v5.1-only configuration
(PR-T-1, PR-T-2), and each child can re-enter Pre-A without accidentally
authorizing implementation because the parent manifest enables only
`-A.proposal`, `G-blocker-1` accepts only the Split verdict, and the parent is a
terminal `framework lifecycle` Split parent with no A–E claim (PR-T-3). No
must-resolve-pre-A unknowns remain (§6).

The **parent therefore ends at Pre-A rather than advancing to Phase A** because
the Split verdict is the framework's terminal route for a deliberately
multi-deliverable program: it authorizes only Phase 0 scaffolding of the six
independent child proposals, trackers, and manifests, and explicitly does not
authorize product implementation, parent Phase A specification, or parent A–E
certification. Every child must separately pass its own non-author Pre-A review
before any Phase A work begins.
