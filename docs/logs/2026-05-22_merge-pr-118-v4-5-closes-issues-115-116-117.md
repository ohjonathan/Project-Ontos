- **Use `gh pr merge --merge --delete-branch` (the project's preferred merge style).** Preserves the full commit history of the lifecycle phases on `main` rather than squashing into a single commit, so future agents can trace which sub-track landed where.

## Alternatives Considered

- **Squash-merge or rebase-merge** instead of merge-commit. Rejected because the 16-commit chain encodes the lifecycle phasing (pre-A → A → B.1 → B.3 → D.2 → D.4 → D.5 → D.6 → E) and the per-issue sub-tracks (`#115` → `#116` → `#117a–f`). Squashing would erase that traceability; the merge-commit topology is the desired final shape.
- **Re-run the strict-P3 dispatch instead of trusting the existing receipts.** Rejected — the receipts are hash-bound in `docs/reviews/project-ontos-github-issues-115-117/lifecycle-receipt-inventory.yaml` and the D.6 gate already validated 12/12 rows including the receipt count and verdict freshness. Re-dispatching would burn worker budget on already-certified work.
- **Promote one of the three residual deferrals to a blocker.** Considered for the doctor JSON key cosmetic delta (`activation_health` vs `activation`). Rejected because the functional contract (non-zero exit on activation hard errors) is verified and tested, the key delta is one-line, and the spec is a living document; cosmetic alignment is a follow-up issue, not a merge blocker.
- **Defer merge until the CLI `ontos activate --json` prose enrichment lands.** Rejected — the reporter's bug surface was the MCP path (which is enriched per contract §2.2.2 and verified at [tools.py:620-640](../../ontos/mcp/tools.py:620)), not the CLI consumer.
