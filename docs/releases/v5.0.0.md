# Ontos 5.0.0 migration guide

Ontos 5.0.0 is the structural release built on the shipped 4.7.1 safety
foundation. It intentionally changes machine-readable CLI results, exit-code
meaning, link locations, command parsing, and rename recovery. No tag, package
publication, or issue closure is implied by this guide.

Release approval is currently **withheld**: product gates pass, but the genuine
strict-P3 lifecycle attempt produced no external verdict receipt. See
`docs/reviews/project-ontos-v5-0-0/final-approval.md` and the hash-bound sibling
evidence index before making any release decision.

## JSON command envelope 4.0

The command envelope moves from schema `3.4` to `4.0`. The domain payload
remains in `data`; the new top-level `result` describes the domain outcome
independently of whether the command executed successfully.

Before:

```json
{"schema_version":"3.4","command":"link-check","status":"success","exit_code":1,"message":"broken references found","data":{"summary":{"broken_references":2}},"warnings":[],"error":null}
```

After:

```json
{"schema_version":"4.0","command":"link-check","status":"success","exit_code":1,"message":"broken references found","result":{"status":"findings","kind":"diagnostic","exit_category":"findings","diagnostics":{"basis":"data.summary","complete":true,"counts":{"broken_references":2}}},"data":{"summary":{"broken_references":2}},"warnings":[],"error":null}
```

Consumers must branch on `status` for execution success, on `result.status`
for clean/findings/warnings/incomplete outcomes, and on
`result.exit_category` when mapping process exits. Do not infer a failed
execution merely because a successful diagnostic command found problems.

For nested commands, the envelope's `command` value is now the canonical
space-separated command path rather than the legacy synthetic hyphenated name:

| 4.7.1 value | 5.0.0 value |
|---|---|
| `mcp-install` | `mcp install` |
| `mcp-uninstall` | `mcp uninstall` |
| `mcp-print-config` | `mcp print-config` |
| `export-data` | `export data` |
| `export-claude` | `export claude` |

Treat `command` as an opaque canonical path, or update exact-match consumers
to the values above.

## Exit codes

| Code | Category | Meaning |
|---:|---|---|
| 0 | `clean` | Operation completed or diagnostic result is clean. |
| 1 | `findings` | Validation or diagnostic findings require attention. |
| 2 | `usage` | Invalid arguments, missing required input, or invalid user input. |
| 3 | `warnings` | Completed with warning-only results, including orphan-only checks. |
| 4 | reserved | Do not assign or depend on this value. |
| 5 | `internal` | Unexpected internal failure. |
| 130 | `interrupted` | Interrupted by the operator or signal. |

Scripts that previously treated every nonzero code as an internal error should
accept `1` and `3` as completed diagnostic outcomes and inspect `result`.
`doctor` warning-only outcomes and `link-check` orphan-only outcomes both use
exit `3`. `query` uses `2` for project/config input failures and `5` for
filesystem or internal failures; those failures never use findings code `1`.

The v5 sweep also removes legacy exit `1` values from execution failures. The
observable command transitions are:

| Command surface | Completed result | Invalid input or refusal | Internal failure |
|---|---|---|---|
| `activate` | validation findings `1`; warnings `3` | missing project/config/docs `2` | map/config I/O `5` |
| `map` | validation or incomplete load findings `1`; strict warnings `3` | missing project/config `2` | unexpected failure `5` |
| `doctor`, `link-check` | findings `1`; warnings/orphans `3` | invalid project/config `2` | unexpected failure `5` |
| `query` | clean query `0` | missing project/config or empty selected scope `2` | document I/O or runtime failure `5` |
| `log` | marker warning `3` | missing repository, unsafe config, or output collision `2` | prepare/write failure `5` |
| `rename`, `retrofit` | clean `0`; retrofit partial warnings `3` | plan/state/format/encoding refusal `2` | commit/recovery failure `5` |
| `env` | parse warnings `3` | invalid path or existing output `2` | write failure `5` |
| `agents`, `export` | clean `0` | existing output or invalid destination `2` | generation/write failure `5` |
| `init` | initialized/already initialized `0`; hook collision warnings `3` | non-Git destination `2` | initialization failure `5`; interruption `130` |
| `scaffold` | dry-run or applied changes `0` | invalid path, malformed content, or refused mutation `2` | discovery I/O, write, or commit failure `5` |
| `mcp install`, `mcp uninstall`, `mcp print-config` | clean or no-op `0` | invalid client, scope, path, or config `2` | config write/render failure `5` |
| `hook --json` | strict validation findings `1`; non-strict warnings `3` | invalid hook input `2` | config/read failure `5` |
| `schema-migrate` | check findings `1`; dry-run warnings `3` | invalid mode or unsupported apply `2` | load/migration/commit failure `5` |
| `promote` | `--check` candidates `1`; no ready documents `3` | noninteractive/refused operation `2` | load/encoding/commit failure `5` |
| `verify` | portfolio discrepancies `1` | invalid path/date/config or interactive `--all --json` `2` | file/database/commit failure `5` |
| `maintain` | task findings `1`; task warnings `3` | workspace/apply guard refusal `2` | task exception or internal operation failure `5` |
| `consolidate`, `stub`, migration export/report | clean `0`; warning-only result `3` where applicable | invalid input, cancellation, or collision `2` | load/write/commit failure `5` |

JSON envelopes for codes `1` and `3` have top-level `status: "success"`; codes
`2`, `5`, and `130` have `status: "error"`. JSON mode suppresses prompts,
progress text, deprecation notices, and human stderr so stdout contains exactly
one envelope.

## CLI parsing and flags

- Global `--json` and `--quiet` work before or after the subcommand. Tokens
  after `--` are positional data and are never reinterpreted as global flags.
- Hidden aliases such as `tree`/`map` and `validate`/`verify` now share the
  canonical command's exact argument contract.
- JSON invocations never prompt. `consolidate --json` and `promote --json`
  select their noninteractive confirmation paths; fresh `init --json` accepts
  hook installation and skips optional scaffolding unless explicitly requested.
- The hidden `hook` command emits one envelope under `--json`. Its normal
  Git-hook invocation retains fail-open exit `0` for non-strict warnings and
  runtime errors so an Ontos configuration problem cannot block Git.
- `env --json` returns the structured `ontos-env-v1` object in `data`, not a
  human report string. Portfolio verification also uses the command envelope.
- Usage failures in JSON mode write exactly one schema-4 envelope to stdout.
  Human-mode error text continues to use stderr.

## Configuration compatibility

Previously accepted numeric configurations remain loadable. Ontos normalizes
unsafe legacy bounds instead of rejecting the project configuration on every
command: `workflow.log_retention_count < 1` becomes `1`, and
`validation.max_dependency_depth < 0` becomes `0`. Consumers that display the
loaded effective configuration should expect those clamped values.

## Link-check locations

Body-reference locations now use physical one-based file lines. Older releases
reported lines relative to the Markdown body after frontmatter. Remove any
consumer-side frontmatter-line offset; editor integrations can navigate to the
reported line directly. Aliased and heading wikilinks are normalized for
resolution while retaining their display alias during rename.

## MCP and rename behavior

MCP read, portfolio, and write tools share one error boundary and preserve
`OntosUserError.code` in the structured error envelope. Portfolio reads also
receive the same pre-activation warning behavior as workspace reads.

`workspace_overview.graph_stats.by_type` is no longer a fixed five-key subset
(`kernel`, `strategy`, `product`, `atom`, and `log`). It is an exhaustive open
map: all canonical ontology types are present with zero-filled counts, and the
`unknown` fallback is retained when an input type cannot be normalized. Its
values now sum exactly to `graph_stats.total`; clients must not assert the old
five-key set.

Before:

```json
{"graph_stats":{"total":8,"by_type":{"kernel":1,"strategy":1,"product":1,"atom":1,"log":1}}}
```

After:

```json
{
  "graph_stats": {
    "total": 8,
    "by_type": {
      "kernel": 1,
      "strategy": 1,
      "product": 1,
      "atom": 1,
      "log": 1,
      "reference": 1,
      "concept": 1,
      "handoff": 0,
      "tracker": 0,
      "retro": 0,
      "review": 0,
      "spec": 0,
      "report": 0,
      "adr": 0,
      "policy": 0,
      "unknown": 1
    }
  }
}
```

CLI and MCP rename now acquire the workspace lock before clean-state checking
and plan construction. Before replacing any file, Ontos writes a durable
byte-for-byte recovery journal under `.ontos/transactions/`. A failed or
interrupted rename restores only touched paths on the next invocation; Ontos
no longer executes an unscoped `git checkout -- .` rollback.

## Adapter checklist

1. Accept `schema_version == "4.0"` and read `result` separately from `data`.
2. Update exit-code handling for findings (`1`), usage (`2`), warnings (`3`),
   internal failure (`5`), and interruption (`130`).
3. Remove body-to-file line offset calculations.
4. Confirm flags intended as positional data are placed after `--`.
5. Parse structured MCP error codes and stop depending on legacy error text.
6. Update exact matches for nested `command` values to canonical paths.
7. Treat MCP `graph_stats.by_type` as an open map and verify counts against
   `graph_stats.total`.
8. Test automation with noninteractive JSON commands before upgrading.

## Documentation validation rebaseline

The kickoff dispatch recorded **35 findings**. Re-running the shipped 4.7.1
`link-check --scope docs` at `454b102` produced 51 broken-reference findings
plus one invalid-status load warning; the difference is scanner/count-basis
drift, not a rewritten historical claim. With the v5 physical-line and
source-file-aware scanner, the repository closes the remaining genuine link
and enum findings. The final v5 validation reports zero broken references,
zero orphans, and zero invalid statuses.
