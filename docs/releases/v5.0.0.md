# Ontos 5.0.0 migration guide

Ontos 5.0.0 is the structural release built on the shipped 4.7.1 safety
foundation. It intentionally changes machine-readable CLI results, exit-code
meaning, link locations, command parsing, and rename recovery. No tag, package
publication, or issue closure is implied by this guide.

## JSON command envelope 4.0

The command envelope moves from schema `3.4` to `4.0`. The domain payload
remains in `data`; the new top-level `result` describes the domain outcome
independently of whether the command executed successfully.

Before:

```json
{"schema_version":"3.4","command":"link-check","status":"success","exit_code":1,"message":"broken references found","data":{"summary":{"broken_references":2}},"warnings":[],"error":null}
```

After:

```json
{"schema_version":"4.0","command":"link-check","status":"success","exit_code":1,"message":"broken references found","result":{"status":"findings","kind":"diagnostic","exit_category":"findings","diagnostics":{"basis":"data.summary","complete":true,"counts":{"broken_references":2}}},"data":{"summary":{"broken_references":2}},"warnings":[],"error":null}
```

Consumers must branch on `status` for execution success, on `result.status`
for clean/findings/warnings/incomplete outcomes, and on
`result.exit_category` when mapping process exits. Do not infer a failed
execution merely because a successful diagnostic command found problems.

## Exit codes

| Code | Category | Meaning |
|---:|---|---|
| 0 | `clean` | Operation completed or diagnostic result is clean. |
| 1 | `findings` | Validation or diagnostic findings require attention. |
| 2 | `usage` | Invalid arguments, missing required input, or invalid user input. |
| 3 | `warnings` | Completed with warning-only results, including orphan-only checks. |
| 4 | reserved | Do not assign or depend on this value. |
| 5 | `internal` | Unexpected internal failure. |
| 130 | `interrupted` | Interrupted by the operator or signal. |

Scripts that previously treated every nonzero code as an internal error should
accept `1` and `3` as completed diagnostic outcomes and inspect `result`.

## CLI parsing and flags

- Global `--json` and `--quiet` work before or after the subcommand. Tokens
  after `--` are positional data and are never reinterpreted as global flags.
- Hidden aliases such as `tree`/`map` and `validate`/`verify` now share the
  canonical command's exact argument contract.
- JSON invocations never prompt. `consolidate --json` and `promote --json`
  select their noninteractive confirmation paths.
- `env --json` returns the structured `ontos-env-v1` object in `data`, not a
  human report string. Portfolio verification also uses the command envelope.
- Usage failures in JSON mode write exactly one schema-4 envelope to stdout.
  Human-mode error text continues to use stderr.

## Link-check locations

Body-reference locations now use physical one-based file lines. Older releases
reported lines relative to the Markdown body after frontmatter. Remove any
consumer-side frontmatter-line offset; editor integrations can navigate to the
reported line directly. Aliased and heading wikilinks are normalized for
resolution while retaining their display alias during rename.

## MCP and rename behavior

MCP read, portfolio, and write tools share one error boundary and preserve
`OntosUserError.code` in the structured error envelope. Portfolio reads also
receive the same pre-activation warning behavior as workspace reads.

CLI and MCP rename now acquire the workspace lock before clean-state checking
and plan construction. Before replacing any file, Ontos writes a durable
byte-for-byte recovery journal under `.ontos/transactions/`. A failed or
interrupted rename restores only touched paths on the next invocation; Ontos
no longer executes an unscoped `git checkout -- .` rollback.

## Adapter checklist

1. Accept `schema_version == "4.0"` and read `result` separately from `data`.
2. Update exit-code handling for findings (`1`), usage (`2`), warnings (`3`),
   internal failure (`5`), and interruption (`130`).
3. Remove body-to-file line offset calculations.
4. Confirm flags intended as positional data are placed after `--`.
5. Parse structured MCP error codes and stop depending on legacy error text.
6. Test automation with noninteractive JSON commands before upgrading.

## Documentation validation rebaseline

The kickoff dispatch recorded **35 findings**. Re-running the shipped 4.7.1
`link-check --scope docs` at `454b102` produced 51 broken-reference findings
plus one invalid-status load warning; the difference is scanner/count-basis
drift, not a rewritten historical claim. With the v5 physical-line and
source-file-aware scanner, the repository closes the remaining genuine link
and enum findings and targets zero broken references, zero orphans, and zero
invalid statuses.
