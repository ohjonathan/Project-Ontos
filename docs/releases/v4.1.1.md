# v4.1.1 — "Stdio Unblock"

**Release date:** TBD (2026-04-12 target)

Patch release fixing a v4.1.0 blocker that prevented every stdio MCP client
(Claude Code, Claude Desktop, Cursor, Continue, Zed, and any `mcp` SDK
consumer) from completing the `initialize` handshake against `ontos serve`.

---

## The Bug

`ontos serve` aliased `sys.stdout = sys.stderr` as a defensive measure during
server bootstrap, then invoked FastMCP's stdio transport inside the same try
block. FastMCP's `server.run(transport="stdio")` snapshots `sys.stdout` at
transport construction time, so every JSON-RPC response for the entire session
was written to stderr. Clients saw an empty stdout, timed out on `initialize`,
and reported the server as unreachable.

Reproduction against v4.1.0 on macOS (Mac Mini M4, pipx-installed
`ontos[mcp]`):

```bash
printf '%s\n' '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"r","version":"0"}}}' \
  | ontos serve --workspace /tmp/ontos-repro \
  1>/tmp/out 2>/tmp/err
# Expected: /tmp/out populated, /tmp/err empty.
# v4.1.0 actual: /tmp/out empty, /tmp/err carries the JSON-RPC response.
```

## Fixes

### Primary — scope the stdout redirect to setup only

`ontos/cli.py` (`_cmd_serve`) now uses
`contextlib.redirect_stdout(sys.stderr)` scoped to the import and
workspace-resolution block. The redirect exits before `serve()` invokes
FastMCP, so the stdio transport binds to the real stdout.

### Secondary — report the correct `serverInfo.version`

`OntosFastMCP.__init__` now accepts a `version` kwarg and assigns it to the
underlying `MCPServer.version` attribute after `super().__init__`. (FastMCP
1.27.0's constructor does not accept or forward `version=`.) `create_server`
passes `ontos.__version__`, so `initialize` responses advertise the installed
ontos version instead of the `mcp` SDK version (`"1.27.0"` previously).

## Regression Guard

`tests/mcp/test_stream_routing.py` adds three subprocess-level tests that
spawn `ontos serve`, send an `initialize` request, and assert:

1. The JSON-RPC response lands on stdout.
2. stderr carries no JSON-RPC payload.
3. `serverInfo.version` matches `ontos.__version__`.

These run under the Python 3.10+ MCP test environment like the rest of
`tests/mcp/`.

---

## Upgrade Notes

- `pipx upgrade ontos` (or `pip install --upgrade ontos[mcp]`) on any machine
  running a stdio MCP client.
- No CLI, config, or MCP tool-signature changes. Non-MCP users are unaffected.

## Breaking Changes

None.

## Acceptance

After upgrading, on the same Mac Mini repro:

```bash
printf '%s\n' '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"r","version":"0"}}}' \
  | ontos serve --workspace /tmp/ontos-repro \
  1>/tmp/out 2>/tmp/err
```

- `/tmp/out` contains a well-formed JSON-RPC response with `id=1` and a
  `result` object.
- `/tmp/err` is empty at the default log level.
- `serverInfo.version` matches the installed ontos package version.
- `claude mcp list` reports `ontos: ... ✓` (green, connected).
