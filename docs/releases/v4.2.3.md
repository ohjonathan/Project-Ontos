# v4.2.3 — "Issue #107 MCP Portfolio Hardening"

**Release date:** 2026-04-15

Patch release closing the residual MCP portfolio hardening items tracked in
Issue #107 after the shipped `v4.2.2` release.

---

## Summary

- `v4.2.2` closed the prior residual hardening slice.
- `v4.2.3` finishes the remaining helper-contract and warning-path cleanup:
  `registry_path` coercion hardening, direct FTS error-contract pinning, and
  stderr resilience for scanner collision warnings.
- This release does not add new CLI commands or MCP tools.

## Fixes

### Portfolio config coercion

`portfolio.registry_path` is now pinned more tightly for programmatic callers:

- missing key still uses the default registry path
- `os.PathLike` inputs are accepted via `os.fspath(...)`
- explicit `None` remains a warning-plus-default fallback
- non-string / non-pathlike values still warn and fall back to the default
- leading and trailing zero-width/BOM edge characters are trimmed in addition
  to the existing whitespace normalization

Interior characters are preserved. Only edge normalization changed.

### FTS invalid-query contract

The invalid-query translation path is now explicitly locked down:

- the SQLite error fragments that map to `E_INVALID_QUERY` are centralized and
  directly tested
- the `"no such column"` branch is pinned as the malformed-explicit-syntax
  path
- plain text remains case-sensitive with respect to explicit FTS keywords

Known limitation: literal-colon queries such as `user:alice` still travel the
explicit-syntax path and remain `E_INVALID_QUERY` in this patch.

### Scanner warning resilience

Slug-collision warnings keep the same stderr format and per-pass re-emission
policy, but warning writes now safely ignore broken-pipe / `EPIPE` failures.

This is resilience hardening only. Slug assignment behavior is unchanged.

## Issue #107 coverage

This release closes the shipped portion of Issue #107:

- **M-D-05** — repeated-call scanner test brittleness
- **M-D-06** — `os.PathLike` / `Path` policy for `registry_path`
- **M-D-07** — explicit `None` semantics for `registry_path`
- **M-D-08** — zero-width edge trimming for `registry_path`
- **M-D-09** — SQLite invalid-query fragment pinning
- **M-D-10** — broken-pipe handling for scanner warning output
- **M-D-11** — direct `"no such column"` translation coverage

## Acceptance

- `registry_path` accepts `PathLike` values and keeps explicit `None` on the
  warning-plus-default fallback
- edge-only zero-width/BOM characters disable registry merging the same way as
  empty strings
- malformed explicit FTS queries stay inside the `E_INVALID_QUERY` envelope
- `user:alice` remains a known limitation and is documented
- scanner collision warnings still emit once per collision per discovery pass
- broken pipe / `EPIPE` on stderr no longer crashes discovery
