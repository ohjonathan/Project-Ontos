# Ontos v5.0.1

v5.0.1 is the patch-safe audit-tail release candidate. It preserves the v5
CLI, schema-4 envelope, and exit taxonomy while removing machine-specific
configuration assumptions, reconciling consolidation retention, hardening the
package provenance gate, and repairing the decision-history format mismatch.

The source metadata is prepared for 5.0.1, but this document does not authorize
or claim a tag, GitHub release, TestPyPI upload, or PyPI publication. Those
release actions remain maintainer-owned.

## Neutral portfolio configuration

New `~/.config/ontos/portfolio.toml` files now contain:

```toml
[portfolio]
scan_roots = []
exclude = []
registry_path = ""

[bundle]
token_budget = 8000
max_logs = 20
log_window_days = 30
```

This is an intentional behavior change. Omitting `scan_roots`, `exclude`, or
`registry_path` in an existing hand-edited config now produces the same empty
value; Ontos no longer substitutes the former machine-specific layout.
`ensure_portfolio_config()` still writes only when the file is absent and never
clobbers an existing configuration.

A writable portfolio server creates the neutral config when needed, then
refuses before creating a database unless at least one scan root or registry is
configured. A read-only server may open an existing database without a config
and continues to avoid config, rebuild, WAL, and SHM writes.
`ontos verify --portfolio` now requires an explicit `registry_path`; it does
not guess a machine-specific fallback.

## Consolidation retention and history recovery

When direct count mode omits `--count`, `ontos consolidate` now reads
`[workflow].log_retention_count`, whose built-in default is 20. An explicit
`--count` has precedence. `--count 0` remains a usage error, and `--by-age`
selection remains independent of the count value. Contributor mode uses the
same built-in workflow default rather than loading a contributor repository's
`.ontos.toml`.

Consolidation also repairs the two-writer format mismatch that could make
`ontos maintain` fail with exit 5. A missing decision-history file is
initialized, and a recognized generated or narrative decision history that
lacks a ledger receives one canonical, six-column `## History Ledger`
section. Repeating the operation does not duplicate the section or an archived
row. A history with an arbitrary identity or malformed existing ledger still
fails closed with zero log moves. The scaffold template now emits the same
canonical table. The broader two-writer architecture remains a v5.x follow-up.

## Shared bundle defaults

The context-bundle defaults now have one named source of truth:

- token budget: 8,000
- maximum logs: 20
- log window: 30 days

The portfolio dataclass, generated TOML, config fallback, and bundler function
defaults all consume those constants. This refactor is behavior-preserving and
does not combine unrelated limits that happen to use the same numbers.

## Exact TestPyPI provenance

The tag-triggered workflow now builds one immutable release bundle. Its
manifest binds the tag and source commit to the wheel and sdist filenames,
kinds, sizes, embedded package versions, and SHA-256 hashes. Every downstream
publication job re-verifies that manifest.

The TestPyPI gate then:

1. verifies the exact published filename/hash set against TestPyPI metadata;
2. downloads only `ontos==<tag version>` from TestPyPI with `--no-deps`,
   `--only-binary`, and `--require-hashes`, without a production-PyPI fallback;
3. compares the downloaded wheel to the release-manifest SHA-256;
4. installs that verified wheel locally with `--no-deps`; and
5. smoke-tests `ontos.__version__`, installed metadata, and the CLI outside the
   checkout.

`skip-existing` is removed, so an occupied TestPyPI version fails closed.
Production publication still depends on successful exact-artifact TestPyPI
verification and consumes the same immutable bundle.

## Upgrade checklist

1. Review `~/.config/ontos/portfolio.toml`. If it relied on omitted keys,
   configure the intended `scan_roots` or `registry_path` explicitly.
2. Add an explicit registry before using `ontos verify --portfolio`.
3. If automation depended on the old direct consolidation default of 15,
   pass `--count 15`; otherwise the workflow retention value now applies.
4. Run `ontos doctor`, then preview repository maintenance with
   `ontos maintain --dry-run`.
