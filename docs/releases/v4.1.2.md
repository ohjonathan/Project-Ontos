# v4.1.2 — "Tools-List Unblock"

**Release date:** 2026-04-13

Patch release fixing a v4.1.1 defect where the `export_graph` tool's
advertised JSON Schema caused strict MCP clients (notably Claude Code)
to silently discard **every** Ontos tool from the `tools/list` response
while still reporting the server as `Connected`.

---

## The Bug

`ontos/mcp/schemas.py` advertised `export_graph.outputSchema` as a root
`{"oneOf": [...]}` discriminated union across its three response shapes
(summary / full / file). The MCP specification requires `Tool.outputSchema`
to describe `CallToolResult.structuredContent`, which is always an
object — a root `oneOf` violates the contract.

Claude Code's MCP client validates the **entire** `tools/list` response
as one Zod schema. A single non-object `outputSchema` caused the whole
response to be discarded, so all 9 Ontos tools vanished from the session
even though the JSON-RPC handshake succeeded. The server reported
`Connected` and logged no errors; tools were simply absent from the
agent's tool list.

Reproduction against v4.1.1:

```bash
printf '%s\n' \
  '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"debug","version":"1"}}}' \
  '{"jsonrpc":"2.0","method":"notifications/initialized"}' \
  '{"jsonrpc":"2.0","id":2,"method":"tools/list"}' \
  | ontos serve --workspace /path/to/any-ontos-workspace --read-only \
  | tail -1 \
  | python3 -c "import sys, json; tools=json.load(sys.stdin)['result']['tools']; [print(t['name'], (t.get('outputSchema') or {}).get('type')) for t in tools]"
# v4.1.1 actual: export_graph prints 'None' (wire-level) and its raw
#   schema dict starts with {"oneOf": [...]}. In Claude Code, all tools
#   are dropped from the session with no surfaced error.
```

## Fixes

### Omit `outputSchema` for `export_graph`

`ontos/mcp/schemas.py::output_schema_for("export_graph")` now returns
`None`. `ontos/mcp/server.py::register()` skips
`set_output_schema` when the schema is `None`, and
`OntosFastMCP.list_tools()` maps a missing entry to `outputSchema=None`
via `dict.get()`. The MCP SDK serializer (`model_dump(by_alias=True,
mode="json", exclude_none=True)`) omits the key entirely on the wire.

Runtime payload validation is unchanged — `EXPORT_GRAPH_ADAPTER` still
validates every `export_graph` response against the same
`Union[ExportGraphSummaryResponse, ExportGraphFullResponse,
ExportGraphFileResponse]` it did in 4.1.1. The tool's three-shape
polymorphic return is preserved; only the advertised schema is
suppressed until a proper object-shaped schema can be designed.

## Regression Guards

Three layers of tests prevent this class of defect from returning:

1. `tests/mcp/test_schemas.py::test_every_advertised_output_schema_is_object_typed`
   — iterates every tool in `TOOL_SUCCESS_MODELS` and asserts each
   advertised `outputSchema` is either `None` or root-typed `"object"`.
2. `tests/mcp/test_server_integration.py::test_list_tools_output_schemas_are_object_or_absent`
   — inspects the `MCPTool` objects returned by `server.list_tools()`
   and asserts the same invariant at the framework layer.
3. `tests/mcp/test_server_integration.py::test_tools_list_wire_payload_has_object_typed_or_absent_output_schemas`
   — transport-level round-trip that mirrors the MCP SDK's exact wire
   serialization at `mcp/shared/session.py:346`
   (`model_dump_json(by_alias=True, exclude_none=True)`), re-parses the
   JSON, and asserts each tool entry either omits `outputSchema` or
   types it as `"object"`. Also pins that `export_graph` has no
   `outputSchema` key on the wire — catching subtle serializer
   regressions (e.g. a future config change emitting
   `outputSchema: null` instead of omitting the key, which some strict
   clients treat as a validation failure).

These run under the Python 3.10+ MCP test environment alongside the
rest of `tests/mcp/`.

## Upgrade Notes

- `pipx upgrade ontos` (or `pip install --upgrade ontos[mcp]`) on any
  machine running a stdio MCP client against `ontos serve`.
- No CLI, config, or MCP tool-signature changes. Non-MCP users are
  unaffected.
- 4.1.1 remains installable but is **not recommended** — it ships the
  defective schema. Consider yanking it from PyPI after 4.1.2 has
  propagated.

## Breaking Changes

None at the MCP tool surface. `export_graph` still returns the same
three response shapes and validates them identically. The only change
visible to clients is that `export_graph` no longer advertises an
`outputSchema` — strict clients now accept the tool instead of
rejecting the entire response.

## Deferred

A proper object-shaped `export_graph.outputSchema` requires refactoring
`ExportGraphResponse` into a tagged object (`mode:
Literal["summary","full","file"]` + corresponding optional payloads).
That is a breaking change for any direct consumer of the raw
three-shape response and is scoped to v4.2.0.

## Acceptance

After upgrading, on the same reproduction workspace:

```bash
printf '%s\n' \
  '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"debug","version":"1"}}}' \
  '{"jsonrpc":"2.0","method":"notifications/initialized"}' \
  '{"jsonrpc":"2.0","id":2,"method":"tools/list"}' \
  | ontos serve --workspace /path/to/any-ontos-workspace --read-only \
  | tail -1 \
  | python3 -c "import sys, json; tools=json.load(sys.stdin)['result']['tools']; [print(t['name'], (t.get('outputSchema') or {}).get('type')) for t in tools]"
```

- Every tool prints `object` except `export_graph`, which prints `None`
  (no `outputSchema` key on the wire).
- No tool prints anything else.
- In Claude Code: `claude mcp list` shows `ontos: ... ✓`, and
  `mcp__ontos__*` tools are available in-session; calling
  `mcp__ontos__workspace_overview` and `mcp__ontos__export_graph`
  returns structured results.
- `~/Library/Caches/claude-cli-nodejs/<session>/mcp-logs-ontos/*.jsonl`
  contains no `Failed to fetch tools` errors.
