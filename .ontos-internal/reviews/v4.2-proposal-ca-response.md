# CA Response: v4.2.0 Proposal Review

**Date:** 2026-04-13  
**Path chosen:** B: Re-Scope

## Critical Issue Decisions

| # | Issue | Decision | Rationale |
|---|-------|----------|-----------|
| C1 | Evidence base | Accept review; narrow scope | The evidence proves the problem class, not the 5-client rollout. Issue [#99](https://github.com/ohjonathan/Project-Ontos/issues/99) established real onboarding failure for Antigravity, the maintainer explicitly asked on 2026-04-13 to apply the same philosophy elsewhere, and current docs already name Cursor / Claude Code / Codex as likely next clients. That is enough to justify continuing the line, but not enough to justify immediate managed automation for five clients. |
| C2 | Contract-drift policy | Managed automation limited to direct JSON contracts with recorded source-of-truth and verified date; drift fails closed | Ontos should not silently rewrite unknown client shapes. For managed adapters, Ontos records the contract source and `last verified` date, validates the expected root key before writing, and on drift returns exit `2` plus the exact `print-config` fallback snippet. Patch target is docs/snippet repair or explicit incompatibility messaging within 72 hours of a reproduced drift report. |
| C3 | CLI delegation | Unify by removing third-party CLI delegation from `v4.2` | The proposal was inconsistent: Claude Code user-scope was rejected while Codex delegated to `codex mcp add`. `v4.2` will not shell out to third-party CLIs at all. Managed automation in this release is only for direct JSON file contracts Ontos writes itself. Claude Code user scope and Codex managed automation both move out of scope. |
| C4 | TOML / Python floor | Keep Python floor unchanged; remove Codex doctor/install from `v4.2` | This is a scope decision, not a runtime blocker. Ontos already carries Python 3.9-compatible TOML read support via `tomli` fallback, so future Codex support remains feasible. It is not justified for this narrowed release. |
| C5 | VS Code | Drop VS Code from first-class managed scope | The proposal did not articulate a principled stability line separating VS Code from Windsurf. Until that exists, VS Code remains in `print-config` / manual guidance only. |

## Open Question Answers

| # | Question | Answer |
|---|----------|--------|
| Q1 | Evidence for demand | Real but weak. The repo has one concrete onboarding defect ([#99](https://github.com/ohjonathan/Project-Ontos/issues/99)), direct maintainer demand to extend the philosophy, and docs already positioning Cursor / Claude Code / Codex as likely next clients. That justifies a narrowed follow-on, not a 5-client managed rollout. |
| Q2 | Claude Code delegation | No in `v4.2`. Claude Code remains `print-config` only; no native CLI delegation in this release. |
| Q3 | TOML / Python floor | No floor bump. Codex managed support is deferred by scope narrowing. Future Codex work can reuse existing `ontos.io.toml` / `tomli` support if the product case becomes strong enough. |
| Q4 | Contract-drift policy | Fail closed, never silent. Managed adapters require a recorded contract source and verified date, validate known JSON roots before writing, and emit `print-config` fallback output on incompatibility. |
| Q5 | Narrow to Cursor only? | Yes. `v4.2` keeps shipped Antigravity support and adds Cursor as the only new managed client. |
| Q6 | Success metric | Within 4 weeks of release, there must be at least one confirmed non-Antigravity real-world use/ask for Cursor onboarding beyond the original Antigravity repro path, and zero contract-drift/config-corruption bugs for the managed adapters. Expansion to another managed client requires one concrete issue/bug/ask for that client or two separate maintainer/user asks plus stable official docs. |
| Q7 | Add `ontos mcp print-config`? | Yes. It becomes the universal zero-write fallback for Antigravity, Cursor, Claude Code, Codex, and VS Code. |
| Q8 | Uninstall / refresh story | `install` remains idempotent and acts as refresh / re-register. Add `ontos mcp uninstall --client {antigravity,cursor}` to remove only the Ontos entry. Document stale-launcher refresh as re-running install. |
| Q9 | Separate roadmap shuffle doc? | Yes. The `v4.2`/`v4.3` sequencing decision should not ride implicitly inside the narrowed onboarding proposal. |
| Q10 | Does VS Code satisfy the stability test? | Not yet for first-class managed scope. It stays in `print-config` / manual guidance only until a separate stability decision says otherwise. |

## Scope Decision

`v4.2` is narrowed to:

- **Managed in `v4.2`:**
  - Antigravity (existing)
  - Cursor (new)
- **`print-config` only in `v4.2`:**
  - Claude Code
  - Codex
  - VS Code
- **Docs-only:**
  - Windsurf
  - Claude Desktop

This preserves the onboarding direction while restoring evidence discipline and limiting contract-drift exposure.
