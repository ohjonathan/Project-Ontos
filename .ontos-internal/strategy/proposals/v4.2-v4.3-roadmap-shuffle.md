# v4.2 / v4.3 Roadmap Shuffle Decision

**Date:** 2026-04-13  
**Status:** Draft decision record  
**Purpose:** Separate the release-sequencing decision from the narrowed `v4.2` onboarding proposal

## Summary

This document records the sequencing decision that was previously
bundled into the `v4.2` onboarding proposal.

The proposal review was correct that these are separate questions:

1. What should `v4.2` build?
2. How should the `v4.x` roadmap be resequenced around that choice?

This document answers the second question.

## Decision

- `v4.2` is reassigned to **MCP client onboarding**, narrowed to:
  - preserve shipped Antigravity support
  - add Cursor as the only new managed client
  - add `print-config` / uninstall lifecycle support
- `v4.3` becomes the next candidate home for:
  - HTTP / Streamable HTTP transport
  - daemon mode / operational hardening
  - security middleware and related transport concerns
  - any future expansion beyond the narrowed `v4.2` onboarding scope

## Why The Shuffle Exists

The old internal roadmap is stale relative to shipped code:

- `v4.1.x` shipped substantially more than the roadmap still claims
- `v4.1.3` proved a real customer-facing onboarding problem
- the immediate product need is onboarding reliability, not transport
  infrastructure

That said, the review was correct that adopting a roadmap reshuffle as a
side effect of the proposal blurred two decisions. This record makes the
sequencing change explicit.

## Status Of The Old `v4.1.1` Backlog

The old `v4.1.1` items are **not** silently absorbed by the narrowed
`v4.2` proposal.

They remain unshipped backlog, specifically:

- search
- `rename_document`
- `promote_document`
- any related contract work that was previously described under the old
  `v4.1.1` plan

This backlog requires separate sequencing after the narrowed `v4.2`
onboarding release is evaluated. It is not implicitly superseded by this
doc.

## Implications

- The main `v4.2` proposal can stay narrowly product-focused.
- Reviewers can assess the onboarding release on its own merits.
- Future planning can decide whether the next release after `v4.2` is:
  - transport / daemon work (`v4.3`),
  - deferred `v4.1.1` backlog,
  - or a different release split entirely.

## Follow-Up

Before implementation planning is finalized:

1. update the canonical `v4_roadmap` document to reflect the chosen
   sequencing
2. explicitly place the old `v4.1.1` backlog in that roadmap rather than
   leaving it implied
3. keep this decision separate from the narrowed onboarding execution
   plan
