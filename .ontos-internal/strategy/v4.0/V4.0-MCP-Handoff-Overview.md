# Ontos v4.0 MCP — Session Handoff Overview

**Date:** 2026-04-04
**Current version:** v3.4.0 (tagged, released)
**Goal:** Implement MCP server as the primary agent interface for Ontos

---

## 1. What is Ontos?

Ontos is a **local-first documentation management system** for AI-assisted development. It solves "context amnesia" — the problem where AI agents lose project knowledge between sessions. Ontos transforms scattered markdown into a persistent, structured knowledge graph (the "Context Map") that serves as shared memory for both humans and AI agents.

**Core loop:** `ontos map` generates a context map → agents read it at session start → `ontos log` records session work → the graph grows over time.

**Package:** `pip install ontos` — Python 3.9+, entry point `ontos.cli:main`.

---

## 2. Why v4.0 / Why MCP?

The strategic vision (documented since v3.0 planning) is to evolve Ontos from a **passive map** (agents read a markdown file) to an **active protocol** (agents query project state dynamically).

> "In v4.0, Ontos will support the Model Context Protocol (MCP), allowing IDEs and Agents to query the project state dynamically ('Ontos, what files are related to the auth module?'). The goal is to become the standard metadata layer for the AI-assisted software development lifecycle."
> — `.ontos-internal/analysis/Ontos-Strategic-Analysis.md`, Section 4.4

**User persona shift:**
- v3.x: Both humans and agents as primary users
- v4.0: Agents primary, humans secondary

## 3. What Exists Today

### 3.1 CLI Commands (potential MCP tool/resource candidates)

| Command | What it does |
|---------|-------------|
| `ontos map` | Generate/update the context map (the core knowledge graph output) |
| `ontos map --compact [plain\|rich\|tiered]` | Token-efficient map variants for agents |
| `ontos log` | Create a session log entry |
| `ontos scaffold` | Auto-tag documents with YAML frontmatter |
| `ontos doctor` | Check graph health (broken refs, orphans, staleness) |
| `ontos query <id>` | Look up a document by ID |
| `ontos link-check` | Scan for broken references, duplicate IDs, orphaned docs |
| `ontos maintain` | Run 9 maintenance tasks (staleness, orphans, promote check, etc.) |
| `ontos rename <old> <new>` | Safe ID rename across the graph |
| `ontos promote` | Promote documents from L0/L1 to L2 |
| `ontos agents` | Regenerate AGENTS.md / .cursorrules |
| `ontos verify` | Verify frontmatter compliance |

### 3.2 Package Architecture

```
ontos/
├── __init__.py          # Version, public API exports
├── cli.py               # CLI argument parsing (511 lines)
├── core/                # Pure business logic — NO I/O (16 files, ~3,800 lines)
│   ├── context.py       # SessionContext
│   ├── graph.py         # Dependency graph
│   ├── frontmatter.py   # YAML parsing
│   ├── ontology.py      # Type definitions
│   ├── validation.py    # ValidationOrchestrator
│   ├── staleness.py     # Staleness detection
│   └── ...
├── commands/            # CLI command handlers (15 files, ~2,400 lines)
├── io/                  # I/O operations — file, git, config, YAML, TOML (6 files)
└── ui/                  # Output formatting — JSON envelopes, OutputHandler (3 files)
```

**Key architectural property:** The `core/` layer is pure logic with no I/O. This was intentionally designed for MCP extraction — the core can be called directly without going through CLI argument parsing.

> See: `.ontos-internal/analysis/Ontos-Technical-Architecture-Map-Codex.md` for the full directory tree with line counts.

## 4. Prior MCP Design Decisions

A 5-LLM review board evaluated MCP strategy during v3.0 planning. Key decisions:

| Decision | Outcome | Reference |
|----------|---------|-----------|
| CLI-first vs MCP-first | CLI-first for v3.x; MCP deferred | `.ontos-internal/archive/v3.0-planning/V3.0-Board-Review-Analysis.md` |
| Python vs Node for MCP | Python (same language as core) | Same, Q12 |
| Pydantic dependency | Only in MCP layer, not core | Same, Q10 |
| MCP security model | Needs definition before shipping | Same, Q11 |
| Auto-configure MCP clients | Deferred to v4.0 | `.ontos-internal/strategy/v3.0/V3.0-Implementation-Roadmap.md`, Section 10 |

### 4.1 Proposed MCP Primitive Mapping (from Chief Architect analysis)

| Ontos Concept | MCP Primitive | Control | Example |
|---------------|---------------|---------|---------|
| Context map | **Resource** | App-controlled | `ontos://project/context-map` |
| Agent instructions | **Prompt** | User-controlled | "Summarize recent decisions" |
| `get_context(focus_ids, depth)` | **Tool** | Model-controlled | Agent fetches context on demand |
| `search_graph(query)` | **Tool** | Model-controlled | Agent searches the knowledge graph |
| `log_session(summary)` | **Tool** | Model-controlled | Agent logs decisions |

> See: `.ontos-internal/archive/v3.0-planning/V3.0-Components/Chief_Architect_v3.0_Comprehensive_Analysis.md`, Part II for the full MCP deep analysis including security considerations.

## 6. Deferred Features (Full List from v3.0 Roadmap)

| Feature | Reason Deferred | Notes |
|---------|-----------------|-------|
| MCP as primary interface | Ecosystem maturity | Now ready to implement |
| Daemon mode | Operational complexity | Long-running MCP server is effectively this |
| Full template system | Scope control | Export templates for different output formats |
| Lazy loading / context slicing | Complexity | `ontos://` URI scheme |
| Auto-configure MCP clients | Depends on MCP | Auto-patch `claude_desktop_config.json`, `.cursor/mcp.json` |
| Cross-repo federation | Enterprise scope | Multi-project graphs |
| S3 archive backend | Local-first philosophy | Out of scope |
| `ontos activate` command | Agent UX debt | Single-command session init |

> See: `.ontos-internal/strategy/v3.0/V3.0-Implementation-Roadmap.md`, Section 10

## 9. Codebase Quick Start

```bash
# Install in dev mode
pip install -e .

# Run the CLI
ontos map                     # Generate context map
ontos map --compact tiered    # Token-efficient variant
ontos doctor                  # Health check
ontos query <doc_id>          # Look up a document

# Run tests (939 passing)
python3 -m pytest -v --tb=short

# Key source paths
ontos/core/       # Pure logic — start here for MCP tool implementations
ontos/commands/   # CLI handlers — reference for what each command does
ontos/io/         # File/git/config I/O
ontos/ui/         # Output formatting (JSON envelopes)
```

## 10. Open Questions for v4.0 Scoping

These decisions haven't been made yet:

1. **Which commands become MCP tools vs. resources vs. prompts?** (Section 4.1 has a draft mapping)
2. **MCP SDK choice** — `mcp` Python SDK (official Anthropic) vs. building on raw JSON-RPC?
3. **Transport** — STDIO only (local), or also HTTP+SSE (remote)?
4. **Security model** — read-only by default? Which tools get write access?
5. **Scope of v4.0** — MCP server only, or also daemon mode / template system / lazy loading?
6. **Auto-configuration** — should `ontos init` set up MCP client configs automatically?
7. **Backwards compatibility** — does CLI remain fully functional, or does it thin out?
