# Ontos v4.0 MCP Proposal: Feedback from Claude Code Architecture Analysis

**Date:** 2026-04-04
**Source:** Analysis of leaked Claude Code source (1,884 TypeScript files, internal codename "Tengu")
**Reviewing:** Ontos-v4.0-MCP-Proposal.md (v3 revision, 508 lines)

## Context

This analysis applies learnings from the leaked Claude Code source to improve both the Ontos v4.0 MCP Proposal and the broader Project Ontos. The key insight: **Claude Code IS Ontos's primary MCP client** -- understanding exactly how its 119KB MCP client connects to, discovers, and processes tool results from MCP servers reveals specific, actionable gaps in the current proposal.

The proposal is well-scoped and architecturally sound. The v4.0.0/v4.1 split is correct. The feedback below focuses on what the proposal gets wrong, misses, or could strengthen -- not on restating what it already does well.

---

## Part 1: Critical Issues (Must Fix Before Implementation)

### 1.1 Tool Annotations Are Absent -- This Will Degrade UX

**The gap:** The proposal defines 6 tools but says nothing about MCP tool annotations. Claude Code actively reads these in `fetchToolsForClient` (client.ts ~line 1796):

```
readOnlyHint: true      -> enables concurrent execution, skips permission prompts
destructiveHint: false   -> no destructive operations  
idempotentHint: true     -> same input = same output
openWorldHint: false     -> no external network calls
```

**Why it matters:** Without `readOnlyHint: true`, Claude Code serializes all Ontos tool calls (one at a time) and may prompt the user for permission on every call. With it, Claude Code runs them concurrently (up to 10 in parallel) and auto-approves. This is the difference between Ontos feeling fast vs. painfully slow.

**Fix:** Add an "Annotations" row to the tool table in Section 4, specifying all four hints for each tool. All 6 tools should have `readOnlyHint: true, destructiveHint: false, idempotentHint: true, openWorldHint: false`.

---

### 1.2 Response Size Will Hit the 25K Token Wall

**The gap:** The proposal specifies no response size constraints. Claude Code enforces a **25,000-token cap** on MCP tool results (`DEFAULT_MAX_MCP_OUTPUT_TOKENS` in `mcpValidation.ts`). Results exceeding this are either truncated with `[OUTPUT TRUNCATED]` or persisted to a temp file (`~/.claude/tool-results/mcp-<server>-<tool>-<timestamp>.txt`), forcing Claude to read it back in chunks.

**Impact on Ontos tools:**

| Tool | Expected Size | Risk |
|------|--------------|------|
| `context_map()` | Ontos-dev map is 87KB (~21K tokens) | **Near cap** -- will exceed for larger projects |
| `get_document()` | Up to 50KB for large docs (Ontos_Manual.md) | **Exceeds cap** -- full content + frontmatter |
| `list_documents()` | ~2KB for 50 docs, ~8KB for 200 docs | Low risk |
| `query()` | ~500B per entity | No risk |
| `health()` | ~200B | No risk |
| `refresh()` | ~100B | No risk |

**Fix:** Three changes to the tool specifications:

1. **`context_map(tier?)` -- add a `tier` parameter** defaulting to `1`. Tier 1 is ~2K tokens (always safe). Agents request Tier 2/3 only when needed. The current design returns all tiers, which will hit the cap for projects with 100+ documents.

2. **`get_document(document_id, content?)` -- add a `content` boolean** defaulting to `true`. When `false`, returns metadata only (frontmatter, word_count, depended_by) without the full markdown body. Agents can browse metadata first, then fetch full content for specific documents.

3. **`list_documents(kind?, status?, offset?, limit?)` -- add pagination**. Return `total_count` in the response. Default `limit` to 100.

### 1.3 Server Instructions Are Missing

**The gap:** Claude Code reads `server.getInstructions()` from every MCP server and includes them in the system prompt (truncated to 2,048 chars per `MAX_MCP_DESCRIPTION_LENGTH`). The proposal does not mention server instructions.

**Why it matters:** Server instructions are "free" context that the LLM always has. Without them, Claude Code has no idea what Ontos is or how to use its tools effectively. It has to guess from tool names and descriptions alone.

**Fix:** Define server instructions (under 2,048 chars) in the proposal. Example:

> "Ontos is a documentation knowledge graph for this project. It tracks architecture docs, strategy decisions, and technical specs as a typed dependency graph. Use `context_map` first to get an overview (Tier 1 is a compact summary, Tier 2/3 provide full details). Use `query` to explore specific documents and their dependency relationships. Use `get_document` to read full document content. All tools are read-only, deterministic, and fast (<100ms cached)."

---

### 1.4 Startup Time Budget Is Unspecified

**The gap:** Claude Code connects to MCP servers with a **30-second timeout** (`getConnectionTimeoutMs()` in client.ts). Local stdio servers are processed in batches of 3 (`getMcpServerConnectionBatchSize()`). If Ontos takes too long to initialize, the connection silently fails.

**What matters:** The proposal says "cold-start cost ~500ms for typical projects" but doesn't treat this as a hard constraint. For Ontos-dev (50 docs), 500ms is fine. For canary.work (46 docs) + role-evaluator-bot (76 docs) in a v4.1 multi-project scenario, graph construction could take 2-3 seconds. Still fine -- but the proposal should state the constraint explicitly.

**Fix:** Add to acceptance criteria: "Server responds to MCP `initialize` handshake within 5 seconds, including cold-start graph construction."

## Part 2: High-Impact Enhancements

### 2.1 Add MCP Resources for Documents

**What Claude Code does:** When an MCP server advertises `capabilities.resources`, Claude Code automatically registers two synthetic tools: `ListMcpResourcesTool` and `ReadMcpResourceTool` (client.ts ~line 2183). These let Claude browse and read resources without the user or LLM needing to know about them in advance.

**What this means for Ontos:** Each Ontos document should be exposed as an MCP Resource with a URI like `ontos://docs/<doc_id>`. This enables a metadata-first browsing pattern:
1. Agent calls `ListMcpResourcesTool` -> sees all documents with metadata
2. Agent selects relevant ones based on metadata
3. Agent calls `ReadMcpResourceTool` -> gets full content for selected docs

**The proposal explicitly excludes Resources** (Section 8, D8: "MCP Resource/Prompt primitives -- v4.0.0 uses Tools only"). This should be reconsidered. Resources are cheap to implement (they map directly to the existing document inventory), and they unlock a browsing pattern that Tools alone don't support well.

**Effort:** Small -- Resources are a thin wrapper over the existing document listing and content retrieval functions.

### 2.2 Add MCP Prompts for Activation Workflows

**What Claude Code does:** When an MCP server advertises prompts, Claude Code fetches them via `prompts/list` and exposes them as slash commands (`fetchCommandsForClient`, client.ts ~line 2033). Users can invoke them directly.

**What this means for Ontos:** The activation protocol currently described in `CLAUDE.md`:
1. Run `ontos map` to generate context map
2. Read `Ontos_Context_Map.md`
3. (At end) Run `ontos log` to record work

...could instead be an MCP Prompt called `/ontos:activate` that does all three steps via tool calls. Similarly:
- `/ontos:maintain` -> weekly 9-task maintenance
- `/ontos:archive` -> end session with log

**The proposal excludes Prompts.** Like Resources, this is a low-effort, high-payoff addition. The CLI commands already exist; Prompts just expose them as discoverable workflows.

### 2.3 Dynamic Tool Descriptions Should Include Workspace Name

**The proposal already mentions this** (D9: "tool descriptions dynamically include the workspace name"). Good -- this is confirmed as a real pattern. Claude Code includes server instructions in its system prompt, and tool descriptions are part of what the LLM sees when deciding which tools to call.

**Enhancement:** Go further. Include document count and last-indexed timestamp in the tool description for `context_map`:

> "Returns the documentation graph for canary.work (46 documents, last indexed 2 minutes ago). Use tier=1 for a compact summary (~2K tokens), tier=2 for the full index, tier=3 for all details including validation."

This gives the agent enough context to decide whether to call the tool, and which tier to request, without needing a separate `health()` call first.

### 2.4 Implement `notifications/resources/list_changed`

**What Claude Code does:** It subscribes to `notifications/tools/list_changed` and `notifications/resources/list_changed` from MCP servers (useManageMCPConnections.ts ~line 620). When a server sends this notification, Claude Code invalidates its cached tool/resource list and re-fetches.

**What this means for Ontos:** When the server detects a file change via the mtime check, it should push a `notifications/resources/list_changed` notification to the client. This lets Claude Code proactively refresh its view of available documents without the agent needing to call `refresh()`.

**This is more important than the `refresh()` tool.** The `refresh` tool requires the agent to know that files changed. Notifications let the server tell the client proactively.

## Part 3: Learnings for the Broader Ontos Project

### 3.1 Context Management Strategies from Claude Code

Claude Code manages context across **6 progressively aggressive strategies** (visible in query.ts):

| Strategy | When | What |
|----------|------|------|
| Tool result budget | Every turn | Cap aggregate tool output size |
| Microcompact | Per result | Inline compression of individual results |
| Snip compact | Configurable | Drop oldest messages from history |
| Context collapse | Token threshold | LLM summarization of old context |
| Auto-compact | Token threshold | Full LLM compaction of entire history |
| Reactive compact | `prompt_too_long` error | Emergency mid-stream recovery |

**Application to Ontos:** The v4.1 context bundle algorithm should adopt this tiered approach. Instead of a single scoring function that tries to optimize a fixed token budget, implement:

1. **Always include:** Tier 1 context map (~2K tokens) -- this is the "tool result budget" equivalent
2. **Include if room:** High-in-degree documents sorted by staleness freshness -- this is the "microcompact" equivalent
3. **Summarize if needed:** For large documents, include a summary rather than full content -- this is the "context collapse" equivalent
4. **Hard cap:** Truncate at token budget with a clear "X documents omitted" notice -- this is the "snip compact" equivalent

This is more robust than a fixed scoring function because it degrades gracefully as projects grow.

### 3.2 The Agent Composition Pattern

Claude Code's `AgentTool` creates sub-agents by cloning the parent context and modifying it:
- Clones the file state cache (shared reads)
- Creates a child AbortController (linked to parent)
- Restricts the tool set
- No-ops state mutations (sub-agents can't affect parent state)

**Application to Ontos:** When Claude Code spawns a sub-agent for a task (e.g., "explore this codebase"), the sub-agent inherits all MCP connections from the parent. This means Ontos tools are available to sub-agents automatically. The implication: Ontos should expect concurrent tool calls from multiple agents within the same session. The asyncio.Lock for concurrent rebuild protection (D4) is correct and necessary.

**Additionally:** Claude Code's coordinator mode (`coordinatorMode.ts`) spawns worker agents that each get access to all MCP tools. Workers share a "scratchpad directory" for cross-worker knowledge. For v4.1, Ontos should consider whether context bundles could serve a similar role -- a pre-assembled knowledge package that a coordinator routes to the right worker.

### 3.3 Memory vs. Knowledge Graph

Claude Code has a file-based memory system (`memdir/`) with:
- `MEMORY.md` as an index (max 200 lines / 25KB)
- Topic-specific memory files with typed frontmatter (user, feedback, project, reference)
- Auto-dream: periodic background consolidation of session transcripts into memories
- Relevant memory retrieval: a lightweight model scores memories against the current conversation

**Ontos IS a more sophisticated version of this pattern.** Claude Code's memory system is per-conversation ephemeral context. Ontos is a persistent, structured, graph-based knowledge system. The key difference: Claude Code's memory is curated by the LLM; Ontos's knowledge graph is curated by the human.

**Strategic insight:** The biggest opportunity for Ontos is not to replicate Claude Code's memory system, but to **complement** it. When Claude Code's auto-dream runs, it could consolidate session decisions into Ontos documents (via `ontos log`). When Claude Code selects relevant memories, it could also query Ontos for relevant architecture context. The MCP server is the bridge that makes this possible.

### 3.4 Configuration Layering

Claude Code uses a 5-source settings system (`utils/settings/constants.ts`):
1. Managed settings (admin/enterprise policy)
2. User settings (`~/.claude/settings.json`)
3. Project settings (`.claude/settings.json`, git-committed)
4. Local settings (`.claude/settings.local.json`, gitignored)
5. Flag overrides (CLI `--settings` flag)

Arrays (like permission rules) accumulate across layers; scalars override.

**Application to Ontos:** The single `.ontos.toml` works for v4.0.0. But for v4.1 multi-project, consider:
- Global: `~/.config/ontos/config.toml` (portfolio-wide defaults: scan roots, token budgets, log retention)
- Per-project: `.ontos.toml` (project-specific overrides: doc dirs, skip patterns)
- The MCP server reads both, with per-project overriding global

This is not a v4.0.0 concern, but the architecture should anticipate it.

## Part 4: What the Proposal Gets Wrong or Contradicts

### 4.1 The Single-Project Model Contradicts the Consumer Brief

**The proposal says** (D9): "v4.0.0 server serves one workspace at a time"

**The consumer brief says** (Consumer 1): JohnnyOS needs `project_registry()` across all workspaces, `get_context_bundle(workspace_id)`, and `search(query_string)` across all projects.

The proposal resolves this by deferring cross-project to v4.1 -- that's fine. But the **deeper problem** is the assumption that Claude Code users will spawn one `ontos serve` process per project.

**What Claude Code actually does:** MCP servers are configured in `settings.json` or `.mcp.json` and spawned once at Claude Code startup. If Ontos is configured as an MCP server, Claude Code spawns ONE process. Running 11 separate Ontos servers (one per documented project) would mean 11 entries in MCP config, 11 stdio processes, and 66 tools. This is technically possible but practically unworkable.

**Better v4.0.0 approach:** Accept the single-project limitation, but document it clearly. The MCP config should specify which project:

```json
{
  "mcpServers": {
    "ontos": {
      "command": "ontos",
      "args": ["serve", "--workspace", "/Users/jonathanoh/Dev/canary.work"]
    }
  }
}
```

**Better v4.1 approach:** A single `ontos serve` process that indexes ALL projects from a portfolio root, with a `project` parameter on each tool to scope queries. This matches Claude Code's expectation of one server = one coherent set of tools.

### 4.2 "CLI Paradox" Framing Understates the Real Value

The proposal's "CLI Paradox" section argues MCP is justified by ecosystem interoperability. This is correct but incomplete. The Claude Code codebase reveals three additional values the proposal doesn't mention:

1. **Schema auto-discovery** (the proposal mentions this). But more importantly: Claude Code's tool deferral system (`ToolSearchTool`) means tools from MCP servers can be lazily loaded. Tools marked with `shouldDefer: true` are only loaded when the agent searches for them. This means Ontos tools could be available but not consume context tokens until needed.

2. **Persistent process state** (the proposal mentions this). But the real value: Claude Code's FileStateCache (`readFileState` in QueryEngine.ts) is an LRU cache of file contents shared across all tools. Ontos's in-memory snapshot serves the same purpose -- and MCP enables it because the server persists across calls, unlike CLI invocations which rebuild on every call.

3. **Tool result processing pipeline.** Claude Code applies microcompaction, tool result budgets, and context management to tool results. MCP results go through this pipeline. CLI output captured via `BashTool` goes through a different, less sophisticated pipeline. MCP tools get better context management treatment than shell-out equivalents.

### 4.3 The `refresh()` Tool Is Less Useful Than Auto-Invalidation + Notifications

The proposal positions `refresh()` as a convenience tool alongside auto-invalidation. But from Claude Code's perspective:

- **Auto-invalidation (mtime check)** is already the primary mechanism -- the agent never needs to call `refresh()` if it works correctly
- **Server-push notifications** (`notifications/resources/list_changed`) are how Claude Code prefers to learn about changes
- **`refresh()`** is only useful if the mtime check fails (which the corrected D4 design says it won't) or if the agent wants sub-millisecond freshness after its own writes (but v4.0.0 has no write tools)

**Recommendation:** Keep `refresh()` but deprioritize it. Implement server-push notifications as the primary freshness mechanism.

## Verification

To validate these recommendations:

1. **Tool annotations:** Configure Ontos as an MCP server in Claude Code, call `context_map()` twice rapidly -- with annotations, both should execute concurrently. Without annotations, the second waits for the first.

2. **Response size:** Call `context_map()` for Ontos-dev without the `tier` parameter. Observe whether Claude Code truncates or persists to file. Then call with `tier=1` and confirm it stays inline.

3. **Server instructions:** After connecting, check Claude Code's system prompt for the Ontos server instructions. They should appear in the model context.

4. **Startup time:** Measure time from `ontos serve` invocation to `initialize` handshake response. Must be under 5 seconds for all documented projects.

5. **Resources:** After connecting, verify that `mcp__ontos__ListMcpResources` and `mcp__ontos__ReadMcpResource` synthetic tools appear in Claude Code's tool list.
