# V4.0-MCP-Proposal-Review-Claude-Code-Analysis-Codex

## Executive Summary

Project Ontos has a clear and durable purpose: it is portable, deterministic, repo-native project memory for AI-assisted development. Its core value is not "documentation management" in the generic sense. Its value is that it preserves architecture, strategy, and decision history in a form that survives tool switching, session resets, rewrites, migrations, and team turnover. That is a real product need, and Ontos addresses it with a stronger philosophy than most agent tooling: explicit structure, human-readable artifacts, and deterministic traversal instead of opaque retrieval.

MCP is therefore a strong fit for Ontos. It converts Ontos from a static artifact model, where agents read `Ontos_Context_Map.md` or shell out to CLI commands, into a queryable context service with a stable programmatic interface. The strategic direction in `Ontos-v4.0-MCP-Proposal.md` is correct: v4.0.0 should be a thin bridge, v4.1 should own portfolio concerns, the filesystem should remain the source of truth, and deterministic retrieval should remain the product philosophy.

The current proposal is still not implementation-safe. It overstates how much existing CLI behavior can be reused directly, underspecifies the actual MCP response contract, and leaves several important client-facing behaviors implicit. The most important corrections are structural, not strategic: stop claiming parity with `ontos map --json`, stop treating `query(entity_id)` as an existing capability, standardize on a canonical snapshot-backed read model, and define v4.0.0 as a new MCP adapter layer over Ontos core and IO primitives rather than as a thin wrapping of existing CLI commands.

This review is based on:

- `README.md`, which defines Ontos as portable context for the agentic era
- `.ontos-internal/strategy/v4.0/Ontos-v4.0-MCP-Proposal.md`
- `.ontos-internal/strategy/v4.0/ontos-usage-and-mcp-motivation.md`
- `.ontos-internal/strategy/v4.0/ontos-v4-mcp-consumer-brief.md`
- `ontos/io/snapshot.py`, `ontos/commands/query.py`, `ontos/commands/map.py`, `ontos/commands/export_data.py`, and `pyproject.toml`
- `claude-code-leak/research/ARCHITECTURE_REPORT.md`
- live CLI inspection performed in the current repo

## What Ontos Is And Why MCP Fits

Ontos is already architected around the right primitives for MCP:

- explicit document types with a clear semantic hierarchy
- deterministic dependency traversal and validation
- human-readable markdown as the source of truth
- a local-first storage model with no external service dependency
- structured exports and reusable graph primitives

That matters because Ontos is strongest when it behaves like a context compiler rather than a chat memory accessory. The README is directionally right: Ontos exists so developers and agents do not have to repeatedly reconstruct project intent from chat history, code archaeology, or vendor-specific memory features. It captures durable project truth, especially at the kernel, strategy, and product layers, in a format that survives implementation churn.

MCP fits that product in three important ways:

- It turns Ontos from a passive file a client must already know to read into an active service the client can discover and query.
- It allows Ontos to hold a warm in-memory view of project state instead of rebuilding from cold process start on every question.
- It exposes Ontos to the broader MCP ecosystem, which is the right strategic frame. The value is not "Claude Code no longer has to shell out." The value is "Ontos becomes the same context service for Claude Code, Cursor, Windsurf, Cline, and future MCP clients."

The proposal should preserve its current good instincts:

- v4.0.0 stays a thin single-workspace bridge
- v4.1 owns cross-project indexing, search, and portfolio routing
- the filesystem remains authoritative for per-project documents
- deterministic retrieval remains the core promise

Those boundaries are not the problem. The problem is that the proposal currently describes the v4.0.0 bridge as simpler and more reuse-driven than the current codebase actually supports.

## Claude Code Learnings That Should Change The Proposal

`claude-code-leak/research/ARCHITECTURE_REPORT.md` shows Claude Code as a real MCP-native client, not a shell-wrapper convenience layer. The report documents a dedicated MCP client subsystem, dynamic tool discovery, MCP resource support, long-lived connection management, and concurrent read-only tool execution. Those architectural facts should directly shape Ontos v4.0.0.

### 1. Add MCP tool annotations for all read-only tools

Ontos v4.0.0 is intentionally read-only. The proposal should say that explicitly in MCP-native terms, not only in prose. All v4.0.0 tools should be annotated as read-only, non-destructive, and idempotent. This better reflects Ontos semantics and makes the MCP contract easier for clients to optimize around.

### 2. Add server instructions

Claude-class MCP clients discover tools dynamically. That is powerful, but tool names alone are not enough grounding. The server should expose a short instruction block that tells the model what Ontos is, how to use `context_map` first, why Tier 1 is the default, and when to fetch documents or resources. This is inexpensive context with disproportionately high value.

### 3. Design for metadata-first access and small defaults

The Claude Code architecture report shows a mature tool-discovery and tool-execution pipeline. Ontos should meet that client with response shapes that browse small and fetch deep only when needed. That means:

- default `context_map` to Tier 1
- allow `get_document` metadata-only mode
- paginate `list_documents`
- expose resources for browse/read flows

This is good MCP design even where the current Ontos repo is not yet close to a hard client ceiling.

### 4. Expose document resources

The architecture report describes MCP resource support as a first-class part of the client model. Ontos documents are a natural resource surface. Tools should remain the main structured query mechanism, but documents should also be browsable as resources such as `ontos://docs/<id>`. That gives clients a clean browse-then-read pattern without overloading tool calls.

### 5. Treat change notifications as first-class freshness behavior

The proposal currently centers `refresh()` plus automatic invalidation. `refresh()` is fine as an override, but the stronger MCP posture is that Ontos should eventually notify the client when the document/resource set changes. v4.0.0 does not need a full watcher-heavy subsystem, but the proposal should at least frame notifications as the preferred steady-state design.

### 6. Design for one long-lived server process as the natural client model

The Claude Code architecture report describes long-lived MCP client connections and additive MCP connections for child agents. Ontos should assume that the typical client experience is one configured server process representing one coherent context service, not repeated command-like invocations. That matters for how the proposal discusses workspace identity, cache lifecycle, and the eventual v4.1 portfolio server.

### What Ontos Should Not Copy From Claude Code

The Claude Code architecture is informative, but Ontos should not imitate parts that are irrelevant to the product or premature for the phase:

- no feature-flag lattice or build-time product slicing for v4.0.0
- no memory or auto-dream imitation
- no agent-orchestration or swarm scope
- no daemon-mode or multi-transport expansion beyond the phase boundary

Ontos should learn from Claude Code's MCP shape, not from its full product surface.

## Must-Fix Problems In The Current Proposal

## 1. The "serialization parity with `ontos map --json`" claim is inaccurate

This is the most concrete error in the document because it is directly falsified by live behavior.

Live inspection result:

- `python3 -m ontos map --json --output /tmp/ontos-context-map-test.md` returned only a success envelope:
  - `command: "map"`
  - `message: "Context map generated"`
  - `data: {"path": "...", "documents": 52, "errors": 0, "warnings": 106}`

It did **not** return the graph, Tier 1 summary, validation structure, or document index that the proposal claims `context_map()` would inherit from `ontos map --json`.

This means the proposal's current wording is not a minor imprecision. It is materially wrong:

- there is no existing `map --json` payload that matches the proposed MCP `context_map()` output
- there is no current "no new serialization layer" path for `context_map()`
- "serialization parity" is not the right framing for v4.0.0

Recommendation: remove the parity claim entirely. Replace it with a new explicit statement: `context_map()` will use a new MCP serializer built from snapshot data and existing map-generation primitives. It may preserve field meanings where sensible, but it is not a pass-through of `ontos map --json`.

## 2. The proposal treats `query(entity_id)` as an existing capability when it is not

The proposal repeatedly frames v4.0.0 as exposing existing capabilities through MCP. That is not true for the proposed `query(entity_id)` tool.

Live inspection result:

- `python3 -m ontos query --help` exposes only:
  - `--depends-on`
  - `--depended-by`
  - `--concept`
  - `--stale`
  - `--health`
  - `--list-ids`

There is no direct entity lookup mode. The current CLI does not support `ontos query <id>` or any equivalent single-document metadata fetch. The proposal therefore invents a tool contract that does not map directly to an existing command.

That does not make the tool invalid in principle, but it does make the current framing misleading. More importantly, the user-requested replacement surface does not need it.

Recommendation:

- remove `query(entity_id)` from the recommended v4.0.0 surface
- let `get_document(document_id, include_content=true)` own single-document retrieval
- let `list_documents(...)` own discovery
- include relationship metadata such as `depends_on`, `depended_by`, and `depth` in `get_document` metadata

This is a cleaner contract and is better aligned with the live codebase.

## 3. "Thin wrapper over existing CLI commands" is overstated

The proposal is directionally right that Ontos has reusable building blocks, but it currently understates the amount of new interface work required.

Live code inspection shows:

- there is no `ontos/mcp/` package today
- `pyproject.toml` has no `mcp` extra and no MCP-related optional dependency
- `rg` over `ontos/` and `tests/` found no actual MCP implementation
- `ontos/io/snapshot.py` already provides `create_snapshot()`, which is the closest existing backbone for MCP reads

That means v4.0.0 is not "activating dormant MCP support." It is adding a new subsystem.

The good news is that Ontos already has the right internal anchor point: `create_snapshot()` plus core graph and validation primitives. The bad news is that this is not equivalent to reusing CLI commands. `map`, `query`, and `export data` have different responsibilities and different payload shapes today. MCP needs a new read adapter layer that sits above snapshot/core, not a command wrapper that sits above argparse.

Recommendation: revise the proposal language from "wrap existing CLI capabilities" to "build a snapshot-backed MCP read layer over existing Ontos core and IO primitives."

## 4. The current invalidation design is still incomplete

The proposal correctly abandons directory `st_mtime`. That is a real improvement. But the replacement is still not a complete cache fingerprint.

The current proposal says the server should compare:

- `max(file.st_mtime)` across tracked files
- file count

That is better than directory mtime, but still insufficient as the full contract because it does not explicitly encode the tracked path set. Path substitutions can evade a max-mtime-plus-count scheme if the replacement preserves the same aggregate signals. This is exactly the kind of edge case that is cheap to eliminate now and expensive to debug once Ontos becomes an external protocol surface.

Recommendation: the fingerprint should be path-set aware. The minimal safe approach for v4.0.0 is to hash a sorted list of tuples such as:

- relative path
- `st_mtime_ns`
- `st_size`

This remains cheap at Ontos scale and closes the design hole without introducing a watcher or database.

## 5. The current codebase has inconsistent document inclusion behavior

This is the most important implementation-readiness issue after serialization.

Live inspection result:

- `python3 -m ontos query --json --health` returned `total_docs: 53`
- `python3 -m ontos export data --json --no-content` returned `summary.total_documents: 52`
- direct inspection of `create_snapshot()` returned `SNAP_DOCS 52`
- direct inspection of the query loader returned `QUERY_DOCS 53`
- `_template` was included in `query` and excluded from `create_snapshot()`

This means Ontos does not currently have one canonical answer to the question "what documents are in the workspace?" That is acceptable inside a CLI with differentiated commands. It is not acceptable once Ontos becomes an MCP server exposing a stable external contract.

The root cause is visible in the code:

- `create_snapshot()` uses config skip patterns
- `query` uses its own scan path via `scan_docs_for_query()`
- the inclusion rules are therefore similar, but not identical

Recommendation: canonicalize document inclusion before MCP ships. The MCP layer should pick one snapshot-backed contract and every read-facing surface should either conform to it or document why it intentionally differs.

## Additional Evidence That Strengthens The Case

`python3 -m ontos export data --json --no-content` currently returns the richest existing structured payload in the repo:

- summary
- documents
- graph
- provenance

That makes `ontos-export-v1` a much better starting point for MCP serialization than `map --json`. It is still not the final MCP contract, because it is export-oriented rather than interaction-oriented, but it is closer to what the proposal wants than the current map command output.

There is also no current evidence that Ontos's own workspace already hits a severe per-call response wall:

- `Ontos_Context_Map.md` is about 22.9k characters, roughly 5.7k estimated tokens
- `docs/reference/Ontos_Manual.md` is about 19.6k characters, roughly 4.9k estimated tokens

That is below the kind of hard budget problem that can break a client immediately. The design conclusion still stands: default-small, metadata-first responses are the better v4.0.0 contract, but the rationale should be framed as MCP ergonomics and future workspace scale, not as a proven current Ontos-dev failure mode.

## Project Ontos Enhancements Beyond The Proposal

The proposal review should not stop at contract fixes. MCP is an opportunity to sharpen Ontos itself.

### 1. Standardize all MCP reads on `create_snapshot()` or a successor canonical snapshot service

`create_snapshot()` is already the closest thing Ontos has to a canonical project read model. MCP should build on that, not on CLI stdout and not on separate command-specific loaders.

### 2. Define one canonical document inclusion policy

Before Ontos exposes itself as a protocol, it needs one explicit answer for:

- whether `_template.md` is ever part of the public document set
- whether `.ontos-internal` is part of the public surface by default
- how `reference` and `unknown` types are handled
- whether internal strategy documents are first-class MCP documents or opt-in library scope

Today those answers are partly embedded in command behavior. MCP requires them to be product decisions.

### 3. Make snapshot-backed serialization a first-class internal API

Ontos needs a serializer layer that turns canonical snapshot state into stable external response objects. That is bigger than MCP. It is the foundation of Ontos as a programmatic interface.

### 4. Treat MCP as the start of a stable API, not just another output mode

The proposal still reads as if MCP is a delivery mechanism bolted onto the CLI. That is too narrow. Once Ontos ships MCP, it has an external contract that other tools can depend on. That contract deserves explicit versioning, serializer tests, and intentional response design.

### 5. Add parity tests around the canonical read model

The right tests are not "MCP equals every existing CLI command." The right tests are:

- snapshot document set is stable and intentional
- MCP serializers consume that snapshot predictably
- CLI commands that claim to represent the same underlying data either match or intentionally differ

This is how Ontos prevents further drift once MCP exists.

### 6. Clarify Ontos's long-term product center of gravity

The best framing is:

- first: project-local context compiler
- second: portfolio index and routing substrate

That sequence matters. It keeps v4.0.0 grounded and prevents Ontos from turning into a vague everything-registry before its local protocol surface is stable.

## Revised v4.0.0 Spec

The current proposal should be replaced with a tighter v4.0.0 contract.

### Recommended v4.0.0 Surface

The v4.0.0 tool surface should be exactly:

1. `context_map(tier=1)`
2. `get_document(document_id, include_content=true)`
3. `list_documents(type?, status?, offset=0, limit=100)`
4. `health()`
5. `refresh()`

Documents should also be exposed as MCP resources, even if prompts remain deferred.

### Why `query(entity_id)` Should Be Removed From v4.0.0

It is the least grounded part of the current proposal:

- it does not map to a live CLI capability
- it duplicates discovery or retrieval responsibilities that fit better elsewhere
- its name is underspecified

If Ontos later wants a structured graph query surface, it should add one intentionally. It should not smuggle it into v4.0.0 under the claim that it already exists.

### Recommended Tool Contracts

#### `context_map(tier=1)`

Purpose:
- provide a compact project overview first
- allow deeper map detail only on request

Contract:
- `tier` is limited to `1 | 2 | 3`
- default is `1`
- response is a new MCP serializer contract built from snapshot data and map-generation logic
- Tier 1 returns summary-oriented content only
- Tier 2 adds the document index
- Tier 3 adds validation and deeper graph detail

This should not claim parity with `ontos map --json`.

#### `get_document(document_id, include_content=true)`

Purpose:
- make single-document retrieval explicit and grounded

Contract:
- requires `document_id`
- `include_content` defaults to `true`
- when `include_content=false`, return metadata only
- metadata includes at minimum:
  - id
  - type
  - status
  - path
  - frontmatter
  - `depends_on`
  - `depended_by`
  - `depth`
  - content hash
  - word count

This tool replaces the proposal's ungrounded `query(entity_id)` behavior.

#### `list_documents(type?, status?, offset=0, limit=100)`

Purpose:
- provide discovery over the canonical document set

Contract:
- supports optional type and status filters
- supports pagination
- returns `total_count` plus page results
- each result should be metadata-first and lightweight

#### `health()`

Purpose:
- identify the connected workspace and current server state

Contract:
- returns workspace name and path
- returns scope
- returns document count from the canonical snapshot
- returns last indexed timestamp
- returns current snapshot fingerprint
- returns Ontos version
- returns server uptime

#### `refresh()`

Purpose:
- force a rebuild when the caller wants immediate certainty

Contract:
- explicitly rebuilds the snapshot
- returns duration and resulting document count
- returns the new fingerprint

Positioning:
- keep it
- do not describe it as the primary freshness mechanism

### Recommended MCP-Native Behavior

The proposal should explicitly recommend:

- all read tools carry read-only and idempotent annotations
- `context_map` defaults to Tier 1
- `get_document` supports metadata-only mode
- `list_documents` is paginated and returns `total_count`
- document resources are exposed even if prompts are deferred
- server instructions explain how to use Ontos effectively

### What v4.0.0 Should Explicitly Reject

The proposal should explicitly defer or reject the following for v4.0.0:

- cross-project registry
- deterministic search
- context bundles
- HTTP/SSE transport
- write-capable tools
- the phrase "no new serialization layer"

## Revised Internal Architecture Recommendation

v4.0.0 should be implemented as a new MCP subsystem over a canonical snapshot service.

### Proposed Responsibility Split

`ontos/mcp/` should own:

- transport bootstrapping
- tool definitions
- resource definitions
- server instructions
- change notifications
- MCP-specific serialization

The snapshot service should own:

- document loading
- filtering
- graph assembly
- validation attachment
- freshness checks
- cache lifecycle

The serializer layer should own:

- mapping snapshot state into MCP response contracts
- separating metadata-first responses from deep content responses
- versioning the MCP response schema independently of CLI envelopes

### Cache Fingerprint Recommendation

Do not ship `max(mtime) + count` as the stated design. Use a path-set-aware fingerprint instead. The simplest acceptable v4.0.0 design is a hash of sorted tuples:

- relative path
- `st_mtime_ns`
- `st_size`

This captures file set changes and cheap file-level change signals without introducing a watcher.

### Concurrency Recommendation

Guard snapshot rebuild with an async lock. Even if the first client path is stdio and mostly sequential, the MCP layer should be explicit about protecting concurrent refresh or stale-check races.

### Why This Must Not Be Built On CLI Stdout

The MCP layer should not parse CLI stdout or reuse command envelopes as its contract surface. The command layer is oriented around human CLI invocation, exit codes, and per-command behavior. MCP is a programmatic protocol surface. It should sit on top of shared internals, not on top of argparse outputs.

## v4.1 Boundary

Keep the v4.1 boundary short and firm:

- portfolio index
- project registry
- deterministic cross-project search
- context bundles
- multi-workspace routing
- possible global config layering

v4.1 should move toward one portfolio-aware Ontos server rather than many manually configured single-workspace servers. That is the model that best matches MCP client expectations and JohnnyOS-style portfolio use.

## Priority-Ordered Recommendations

| Priority | Recommendation | Why it matters | Phase |
|---|---|---|---|
| P0 | Fix the `ontos map --json` serialization claim | The current proposal makes a false claim about live behavior and understates the amount of new serializer work | v4.0.0 |
| P0 | Replace the `query(entity_id)` assumption | The proposed tool does not map to a real current capability and muddies the v4.0.0 surface | v4.0.0 |
| P0 | Canonicalize snapshot and loader behavior | Ontos cannot expose a stable MCP contract while different read paths disagree on the document set | v4.0.0 |
| P1 | Add read-only annotations and server instructions | This improves MCP client grounding and better reflects Ontos semantics | v4.0.0 |
| P1 | Add Tier 1-by-default and metadata-first response shaping | This makes the interface more usable and scales better than full-payload defaults | v4.0.0 |
| P1 | Add document resources | Resources are a natural fit for Ontos documents and improve browse/read flows | v4.0.0 |
| P1 | Improve the invalidation fingerprint | `max(mtime) + count` is still incomplete for an external protocol surface | v4.0.0 |
| P2 | Keep portfolio features deferred | This preserves the thin-bridge discipline and prevents v4.0.0 from absorbing unvalidated scope | v4.1 |

## Final Assessment

Project Ontos is pointed in the right direction. The product thesis is strong, the architecture is compatible with MCP, and the v4.0.0 versus v4.1 split is the correct strategic move. The proposal does not need a strategic rewrite.

It does need a technical rewrite in a few critical places. The current draft overclaims existing CLI reuse, describes one tool that does not actually exist, and leaves the canonical read model too ambiguous. Those are fixable problems. The right answer is not to back away from MCP. The right answer is to tighten the proposal around a snapshot-backed MCP read layer, reduce the v4.0.0 surface to what is truly grounded, and treat MCP as the beginning of Ontos's stable programmatic interface.
